# オーケストレーター判断基準

役割と手順の正本は `AGENTS.md` / `.cursor/rules/orchestrator.mdc` / `docs/workflows/orchestrate.md`。
このスキルは、正本が「オーケストレーターの見立て・判断」に委ねている箇所の**判断基準だけ**を具体化した補助。手順はここには書かない。**正本と矛盾したらこのスキルが間違っている**(正本に従い、気づいたらこのスキルを修正する)。

初版はテンプレート構造から起こした既定値。実プロジェクトで判断を誤った事例が出るたびに追記して育てる(末尾「このスキルの更新」)。

## 1. 難易度見立てのシグナル(モデル・effort 決定の前に)

正本の3段階(軽量 / 標準 / 高難度)を、次のシグナルで判定する。難易度 → モデル・effort の対応は正本の表(orchestrate.md「モデル・effort の自動判断」)で決める。

**高難度シグナル(1つでも該当 → 高難度)**

- 状態・時間・金・権限のいずれかを扱う(認証、決済、セッション、権限分岐、並行処理、タイムゾーン)
- UI + API + DB など複数レイヤーを跨ぎ、状態整合・契約変更・失敗時復旧を同時に考える変更。**ファイル数だけでは高難度にしない**(通常の機能追加でも3ファイル以上になるため)
- 既存データの変換・移行を伴う
- 外部 API を2つ以上組み合わせる
- 同じタスクの2回目以降の再委譲(1回失敗している)
- 仕様全体の整合レビュー・セキュリティ観点のレビュー

**軽量シグナル(すべて該当 → 軽量)**

- 変更が1ファイルで、直し方に選択の余地がない(typo・リンク・設定値1つ)
- 失敗してもすぐ気づけて、git で即戻せる

どちらでもなければ標準。迷ったら一段上(正本)。ただし「なんとなく不安」だけでは上げない — 上げるなら該当シグナルを1つ言えること。

## 2. 委譲プロンプトの送信前チェック

送信直前に自問する。1つでも No なら書き直す(1分の書き直しの方が、要修正 → 再委譲の1往復より安い)。

- 会話文脈ゼロの相手が、このプロンプト**だけ**で完遂できるか(「さっきの」「例の」等の指示語が残っていないか)
- ① 読むファイルをパスで列挙したか(該当 workflow + 対象ファイル + 上流ドキュメント)
- ② 出力先をパス + ファイル名形式(`YYYY-MM-DD_<slug>.md`。slug は ASCII の英小文字・数字・ハイフン)まで指定したか
- ③ 前提に「確定済みの決定事項」と「やらないこと(スコープ外)」を書いたか
- ③ 前提に「実行指定AI / モデル / effort」の1行を入れたか。**委譲コマンド自体にもモデル・effort を明示したか**(Codex: `-c model_reasoning_effort=…` / Claude: `--model … --effort …`)。いずれも見立てた難易度と一致しているか(正本の必須事項)
- 「質問はできない。判断できない点は未決事項に書いて完走する」を含めたか
- 完成条件(何が書けていれば完了か)を1行で書いたか

## 3. 止まる / 自律判断 / 進む の切り分け

迷ったら三択で考える。

**(a) 止まる** — 正本の STOP 条件とユーザー承認ポイントに該当するときだけ。境界の既定解釈:

- 「新しいライブラリの導入」= 直接依存への追加。既存依存の推移的依存や `@types/` 等の型定義だけなら該当しない(判断に迷う規模なら聞く)
- 「破壊的操作」= 戻す手順が自明でない操作(force push・履歴改変・DB スキーマ変更・一括削除)。git で戻せる通常のファイル編集は該当しない
- 「同じ問題を2回以上修正しても通らない」= 同じエラー・同じ症状が続くこと。**別のエラーに変わったら前進とみなし、カウントをリセットする**
- 「費用が発生する操作」= 新規の課金・契約・従量 API の追加。再デプロイ自体に新規費用がなくても、`docs/status.md` で自動許可されていなければ停止する

**(b) 自律判断して記録** — 次の3つを満たすなら決めて進み、自律判断ログか `docs/status.md` の決定事項ログに1行残す:

- 後から戻すのが容易(命名・構成・軽微な選択)
- どちらを選んでも上流ドキュメントと矛盾しない
- `docs/decisions/` の過去の決定から方向を推定できる(無ければ最小変更で安全な側)

**(c) そのまま進む** — 正本に手順が定義済みの定型作業。判断ですらないものに確認を求めない(Cursor の Shell 承認は UI の権限確認 — 正本どおり承認して続行)。

## 4. レビュー判定のエッジケース

原則(判定直下1行だけ読む・本文から再判定しない・条件付き承認 / 要修正は Codex に反映委譲・最大2ラウンド)はすべて正本にある。ここはレポートが崩れていた場合だけ:

- `## 判定` セクション自体が無い → 成果物欠落として扱い、同じ委譲を1回だけ再実行。再度欠けたら STOP(正本の欠落条件)
- 判定行が「承認 / 条件付き承認 / 要修正」以外の文言 → 推測で読み替えず、規定の3値で出し直すよう同じ委譲を1回だけ再実行。再度不正なら STOP
- 本文の指摘の重さが判定と釣り合って見えなくても、判定に従って進める(正本)。懸念があれば最終応答に1行だけ書く
- 反映委譲では指摘を自分で要約・取捨選択せず、レポートのパスを渡す(正本のテンプレどおり)

## 5. 成果物検証の最低ライン

「実体を開いて確認・鵜呑みにしない」(正本)の具体化。委譲完了ごとに:

| 成果物 | 最低限の確認 |
|---|---|
| 要件定義書・仕様書 | テンプレの必須セクションが埋まっている / 上流へのリンクがある / 未決事項欄が存在する(空でも可) |
| 実装 | lint・ビルド・既存テスト全件を実際に実行 / 作業報告に書かれたファイルが実在する / lockfile・依存・秘密情報を確認 / 委譲したマイルストーン範囲外の変更が混ざっていない |
| レビュー | `## 判定` 直下1行が規定の3値で読める |
| 反映 | 指摘と修正箇所の対応がレポートのチェックボックス等で追える |

スコープクリープ検知: `git status` と diff で、委譲範囲と無関係なファイルの変更を見る。混ざっていたら自分で直さず、差分を確認してユーザーに報告のうえ要修正として差し戻す(無断で握りつぶさない — 正本「委譲先の暴走」)。

## 6. リカバリの共通原則

個別の対処(ハング・`xhigh` 非対応・未認証)は正本にある。ここは共通の考え方だけ:

- **リトライ予算は各症状につき1回**。2回目の失敗で取るのは「別のアプローチ」か「STOP」であり、同じコマンドの3回目ではない。例外はデプロイ失敗のみ(正本どおり原因調査・修正を2回まで試みてから STOP)
- 出力ファイルが空・存在しない場合、まず自分の委譲プロンプト(出力先パス・3点セット)の不備を疑う。直して1回だけ再実行
- 復旧のために作成作業を肩代わりしない(正本)。「委譲し直す / 分割する / STOP」から選ぶ

## 7. このスキルの更新

- オーケストレーターの判断ミス(不要な停止・止まるべきで止まらなかった・難易度の見誤り・委譲プロンプト不備による手戻り)に気づいたら、事例を1行で `docs/decisions/` か `docs/status.md` の決定事項ログに残す
- 事例が数件たまったら、上位モデル(claude.ai の最上位モデル等)にこのスキル + 事例を渡して改定案を作らせ、テンプレートに還元する(CHANGELOG に記録)
- 正本(orchestrate.md 等)の改定でこのスキルと矛盾が出たら、正本に合わせてこちらを直す
