## 1. 概要

- 目的: スマホWebアプリで、ゴミ様の対象物をカメラ認識し、認識結果から音とビジュアル作品を生成・再生し、持ち帰り再生までの成立性を検証する
- 位置づけ: 本番ツアー運用ではなく、技術ハードルと体験価値を確認する検証用デモ
- ターゲットユーザー:
  - 体験者: 藤クリーンの産廃ツアー参加者、関係者向けデモ参加者
  - 運営者: デモ当日に対象物、NFCタグ、QR、撮影台を準備・補助する担当者
- 成功の定義:
  - スマホブラウザでカメラを起動し、用意した対象物を1種類以上認識できる
  - 認識結果をシードにして、音とビジュアルが端末内で生成・再生される
  - 生成結果が同じシードから再現でき、URL / QR / NFC のいずれかで別端末または後日の再生導線を確認できる
  - マーカーレスARが難しい場合でも、NFC、QR、画像ターゲット、レーザー彫刻パターンのいずれかで「物体から作品を呼び出す」代替導線を提示できる

## 2. スコープ

### やること(In)

- スマホブラウザで背面カメラを起動する
- 運営側が用意した少数のゴミ様対象物を認識する
- 認識結果から作品シードを生成する
- 作品シードをもとに、p5.js相当の生成ビジュアルと Tone.js 相当の生成音を端末内で再生する
- 生成結果をURLパラメータで再現できる形にする
- 最小デモでは、PNG保存または共有用URLコピーまでを実装する
- 段階追加として、NFCタグへのURL書き込み、QR表示・読み取り、画像ターゲットAR、レーザー彫刻パターンを検証する
- iOS Safari / Android Chrome のスマホ実機で主要フローを検証する

### やらないこと(Out)

- 本番ツアー運用の予約、受付、参加者管理
- ユーザー認証、アカウント、個人別ギャラリー
- 課金API、商用生成AI API、外部サーバーでの大規模生成
- 任意の未知ゴミを高精度に分類する汎用ML
- 大規模ML学習パイプライン、MLOps、継続学習
- ネイティブアプリ前提のVuforia等による本格3D物体追跡
- 個人情報、顔、位置情報を用いる体験設計
- 本番公開、外部公開、Git push、デプロイ設定の確定

|---|---|---|
| 開始画面 | カメラ・音声のユーザー操作起点を作る | 開始ボタン、カメラ許可案内、音声開始処理、対応ブラウザ注意 |
| カメラ認識画面 | 対象物を写して認識状態を確認する | カメラプレビュー、認識ラベル、信頼度、撮影/生成ボタン、認識エラー表示 |

## 4. 機能仕様

### 4-1. カメラ起動

- 挙動:
  - 開始ボタン押下後、`getUserMedia` で背面カメラを要求する
  - HTTPSまたはlocalhost上でのみ動作対象とする
  - iOS向けに動画要素は `playsinline` / `muted` を前提にする
- 入力/出力:
  - 入力: ユーザー操作、カメラ許可
  - 出力: カメラ映像ストリーム、エラー状態
- エッジケース:
  - カメラ拒否: 再許可方法とブラウザ再読み込みを案内する
  - アプリ内ブラウザ: Safari / Chromeで開き直す案内を出す
  - 背面カメラが取れない: 利用可能カメラにフォールバックする

### 4-2. 対象物認識

- 挙動:
  - M1では、用意した対象物1種を認識対象とする
  - M2以降で、実物候補と撮影条件が揃った場合に3〜5種へ増やす
  - 推奨は Teachable Machine で作成したTF.js分類モデル、またはTF.jsの軽量分類モデルを利用する
  - 分類モデルには対象物クラスに加えて背景/非対象クラスを1つ用意する。用意できない場合は信頼度閾値で非対象を判定する
  - 認識結果は、ラベル、信頼度、認識時刻、モデルバージョンとして保持する
- 入力/出力:
  - 入力: カメラフレーム
  - 出力: `objectLabel`、`confidence`、`modelVersion`
- エッジケース:
  - 信頼度が閾値未満: 未認識として扱い、撮影条件の調整を促す。M1実装時に実機の誤認識を見て暫定閾値を決め、検証記録に残す
  - 複数候補が近い: 上位候補を表示し、M1では最上位のみ採用する
  - 端末が遅い: 推論頻度を落とし、手動撮影モードに切り替える

### 4-3. 作品シード生成

- 挙動:
  - 認識ラベル、信頼度レンジ、撮影ごとの簡易画像特徴またはランダムnonce、生成バージョンから短いシードを作る
  - 規則性は「同一シードなら同一の音・ビジュアルを再生成できる」ことで担保する
  - 偶然性は「同じ対象物を撮影しても、撮影ごとに別シードになってよい」ことで担保する。M1.5では同一対象物から常に同一シードが出ることは要求しない
  - シードには生成ロジックのバージョン番号を含める
- 入力/出力:
  - 入力: 認識結果、任意のユーザー操作、撮影ごとの簡易画像特徴またはnonce、生成バージョン
  - 出力: `seed`, `generatorVersion`, `params`
- エッジケース:
  - シードが壊れている: 再生画面でエラー表示し、デフォルト作品を出さない
  - 旧バージョン: 旧生成ロジックを残すか、再生成差分を明示する

### 4-4. 音生成・再生

- 挙動:
  - ユーザー操作後にAudioContextを開始する
  - シードから音階、BPM、音色、ノイズ量、ループ長を決定する
  - 再生/停止、音量調整を用意する
- 入力/出力:
  - 入力: `seed`, `soundParams`
  - 出力: ブラウザ内の音声再生
- エッジケース:
  - 音が鳴らない: 再生ボタンを再提示し、ミュート状態を表示する
  - 端末負荷が高い: 同時発音数とエフェクトを減らす

### 4-5. ビジュアル生成

- 挙動:
  - Canvas上に、シード駆動のジェネレーティブビジュアルを描画する
  - M1では静止画または軽いループアニメーションを対象にする
  - PNG保存を実装する
- 入力/出力:
  - 入力: `seed`, `visualParams`
  - 出力: Canvas表示、PNG Blob
- エッジケース:
  - 低性能端末: 解像度を下げる
  - 保存不可: 共有URLコピーを代替導線にする

### 4-6. 持ち帰り再生

- 挙動:
  - 作品シードをURLパラメータ化する
  - URLから開いた再生画面で同じ音・ビジュアルを復元する
  - M2では共有URLとQRを検証する。QRの読み取りはOS標準カメラまたは標準QR読み取り機能を前提とし、アプリ内QRリーダーはM2では作らない
  - M2.5ではNFCタグへのURL書き込みをAndroid Chromeまたは既製NFC書き込みアプリで検証する
  - iPhoneでは、事前にURLを書き込んだNFCタグをOS標準の読み取りで開く運用を想定する
- 入力/出力:
  - 入力: `seed`入りURL、QR、NFCタグ
  - 出力: 再生画面、同一作品
- エッジケース:
  - NFC非対応: QRまたはURLコピーへフォールバック
  - iOSでWeb NFC書き込み不可: Android端末または既製NFC書き込みアプリで運営側が書き込む
  - オフライン: PWAキャッシュが済んでいない場合は再生不可として扱う

### 4-7. AR / 代替AR

- 挙動:
  - M3では、マーカーレス物体追跡ではなく、画像ターゲットまたは彫刻/印刷パターンを追跡対象にする
  - MindAR等の画像ターゲット方式で、物体上にCanvasまたは軽量3D表現を重畳する
  - 画像ターゲットが不安定な場合は、QR / NFC / model-viewer型の空間配置へ切り替える
- 入力/出力:
  - 入力: カメラ映像、ターゲット画像、`seed`
  - 出力: AR重畳表示、または通常再生画面
- エッジケース:
  - ターゲットを見失う: 通常再生に戻す
  - 彫刻コントラスト不足: 印刷ラベルまたはNFCへ切り替える
  - WebXR非対応: WebXR前提の体験にしない

## 5. 実現性・ハードル評価

| 項目 | 評価 | 仕様上の扱い | 主なリスク |
|---|---|---|---|
| 画像認識 | 低 | 用意した対象物の分類に限定し、M1から採用する | 照明、背景、対象物の類似、学習データ不足 |
| マーカーレスAR | 高 | 本要件からは外し、画像ターゲット / NFC / QR / model-viewerへ分解する | iOS SafariのWebXR制約、任意物体の再訪追跡がWebでは難しい |
| デバイス内生成 | 低 | p5.js相当 + Tone.js相当のシード駆動生成をM1から採用する | 音声開始にユーザー操作が必要、端末負荷 |
| 持ち帰り再生 | 低〜中 | URLシードを主方式、NFC/QRを段階追加する | iOSでWeb NFC書き込み不可、タグ素材・貼付方法、オフライン再生 |

補足:

- 調査レポートでは、ブラウザ内diffusion等の重量級生成はスマホでは非現実的と評価されているため、本仕様では採用しない
- 「ゴミ様物体にマーカーレスARで作品を記録する」は、Web上の物体追跡としては高難度のため、「作品シードを物体に紐づけ、NFC/QR/画像ターゲットで呼び出す」に読み替える
- レーザー彫刻は、QRまたは画像ターゲットの物理パターンとして検証する。成立性は素材・コントラスト・線幅に依存するためM3扱いにする

## 6. 非機能要件

- 対応デバイス:
  - 優先: iPhone Safari、Android Chrome
  - 開発確認: PC Chromeのlocalhost、スマホ実機のHTTPS
- HTTPS配信手段:
  - 開発中のPC確認は `localhost` を使う
  - 同一LAN内のスマホ実機確認は、mkcert等でローカルHTTPS証明書を用意し、開発PCのLAN IPへ接続する方式を第一候補にする
  - 同一LAN方式で端末・証明書設定が詰まる場合は、一時トンネルまたは限定公開テストURLを候補にする
  - 外部到達可能なURLを使う場合は、本番公開ではなく検証用の限定公開として扱い、可否と方式をユーザー承認後に決める
- パフォーマンス:
  - M1の認識応答は、撮影または生成操作から3秒以内を目標にする
  - ビジュアル生成は初回表示3秒以内、再生中は体感で操作不能にならないこと
  - 推論は常時30fpsを必須にせず、低速端末では手動撮影・低頻度推論を許容する
- アクセシビリティ:
  - 音声が出ない環境でもビジュアルで結果が分かる
  - 主要ボタンはタップしやすいサイズにする
  - 認識結果は色だけでなくテキストでも表示する
- プライバシー:
  - カメラ映像は端末内処理を基本とし、M1〜M3ではサーバー送信しない
  - 個人情報、顔、位置情報を保存しない
- 運用:
  - デモ前に対象物、撮影台、照明条件、NFCタグまたはQRを準備する
  - オフライン利用を想定する場合は、PWAキャッシュ済み端末を事前確認する

## 7. 技術構成

- 種別: Webアプリ(SPA)
- 想定アプリ名: `camera-ar-gomi-demo`
- 想定配置: `apps/camera-ar-gomi-demo/`

### 技術候補比較

| 候補 | 内容 | 長所 | 短所 | 評価 |
|---|---|---|---|---|
| A. Vite + TypeScript + TF.js + p5.js + Tone.js | 静的SPAとして実装 | 既存テンプレ規約に合う。GitHub Pages等へ載せやすい。オンデバイス完結 | ARは別ライブラリ追加が必要 | 推奨 |
| B. 素のHTML/CSS/JS + CDN | 最小構成で実装 | M1の試作が最速。ビルド不要 | M2以降で構成が散らばりやすい | 超短期試作用 |
| C. React + Vite | UI状態管理をReactで実装 | 画面遷移と状態管理が整理しやすい | Canvas/カメラ/音声中心の小規模デモでは過剰になりやすい | 必要なら採用 |
| D. 8th Wall系 / XR Engine | WebAR基盤をセルフホスト前提で検証 | 無料化後の構成なら外部ホスト依存を下げられる可能性がある | 新体制直後でライセンス、保守性、実装負荷の確認が必要 | 下書き段階では不採用 |
| E. Zappar等の商用WebAR | ARを商用SDKに任せる | iOS/Android横断のAR品質が高い | 課金・契約・外部サービス判断が必要 | 下書き段階では不採用 |
| F. ネイティブアプリ | ARKit/ARCore/Vuforia等を利用 | 物体追跡の選択肢が広い | 工数が大きくWebデモの範囲外 | 不採用 |

### 推奨案

- Vite + TypeScript + 静的SPA
- 画像認識: TF.js + Teachable Machineエクスポートモデルを第一候補
- ビジュアル生成: Canvas / p5.js
- 音生成: Web Audio API / Tone.js
- 持ち帰り: M2でURLシード + QR、M2.5でWeb NFC(Android Chrome)とNFCタグ読み取り運用を追加
- AR: M3でMindAR画像ターゲットを検証。WebXR必須にしない
- PWA: M2以降でService Workerを追加し、アプリ本体とモデルをキャッシュする

選定理由:

- M1で必要なカメラ、認識、生成、保存がすべて静的Webアプリで成立する
- サーバー、課金API、ユーザー認証を避けられ、検証用デモとしてリスクが低い
- 作品をシードで表現すれば、NFCやQRに短いURLとして持たせられる
- WebXRや本格物体追跡を前提にしないため、iOS/Android差分に引きずられにくい

## 8. データ(Webアプリの場合)

### データモデル

| 名称 | 内容 | 保存先 |
|---|---|---|
| `RecognitionResult` | ラベル、信頼度、モデルバージョン、認識時刻 | メモリ |
| `ArtworkSeed` | シード文字列、生成バージョン、対象物ラベル、パラメータ | URLパラメータ、必要に応じてlocalStorage |
| `GeneratorParams` | 色、形、速度、音階、BPM、音色 | シードから復元 |
| `ShareLink` | 再生画面URL、QR用文字列、NFC書き込み用URL | URL / QR / NFC |
| `ModelManifest` | 認識モデルのURL、ラベル一覧、バージョン | アプリ同梱またはpublic配下 |

### 保存方針

- 主保存: URLパラメータにシードを含める
- 補助保存: localStorageは直近の再表示用に限定する
- ファイル保存: PNG保存をM1、WAV/音声書き出しはM2以降の任意扱い
- サーバーDB: M1〜M3では使わない

## 9. マイルストーン

| # | 内容 | 完了条件 | 検証方法 |
|---|---|---|---|
| M1 | 最小縦切り: カメラ起動 + 1対象物認識 + 1反応 | スマホのHTTPS環境でカメラが起動し、用意した対象物1種を認識して、Canvas表示または音のどちらか1つが変化する | iPhone Safari / Android Chromeの少なくとも一方で、開始→許可→対象物を写す→認識ラベル表示→反応確認までを連続操作する |
| M1.5 | 音・ビジュアルの両方を生成 | 1回の生成で得たシードから、音とCanvasが同時に再生/表示され、同一シードで同じ見た目・音のパラメータが復元される | 生成したシードURLを再読み込み、可能なら別端末でも開き、同一出力が復元されることを確認する |
| M2 | 共有URL / QR持ち帰り | シード入りURLを開くと、別端末または再読み込み後に同じ作品が再生される。QRから同じURLを開ける | 生成画面でURL/QRを作成し、別スマホで読み取って再生画面を開く |
| M2.5 | NFC持ち帰り | Android Chromeまたは既製アプリでNFCタグにURLを書き込み、スマホでタグを読むと再生画面が開く。iPhoneでの読み取り確認を含める。Androidのみで完了する場合は、iPhone未確認である旨を検証記録に残す | NFCタグにURLを書き込み、iPhoneで読み取り→再生を確認する。Androidでも書き込み/読み取りを確認し、iOSでWeb書き込みできない点は運用手順に記録する |
| M3 | 画像ターゲットAR | 印刷またはレーザー彫刻想定の画像ターゲットをカメラで認識し、対象上に作品ビジュアルを重畳できる | テストターゲットを用意し、スマホでターゲット検出→重畳表示→見失い時の通常再生フォールバックを確認する |
| M3.5 | レーザー彫刻 / 物理パターン試作 | 実物素材に作ったQRまたは画像ターゲットを読み取り、持ち帰り再生またはAR重畳のどちらかが成立する | 素材別に彫刻サンプルを作り、読み取り成功率、距離、照明条件、失敗時のNFC代替を記録する |

## 10. 代替案・フォールバック

| 難所 | 第一案 | フォールバック | 採用条件 |
|---|---|---|---|
| 対象物認識が不安定 | TF.js分類 | QR / ARマーカーで個体IDを読む | 認識成功率がデモ条件で8割未満の場合 |
| マーカーレスARが成立しない | 画像ターゲットAR | NFC起動 + 通常再生、QR起動 + 通常再生、model-viewer空間配置 | iOS/Android両対応を優先する場合 |
| NFC書き込みが端末依存 | Android Web NFC | 既製NFC書き込みアプリ、QRカード | 運営端末を固定できない場合 |
| レーザー彫刻の読み取りが不安定 | 彫刻QR / 彫刻画像ターゲット | 印刷ラベル、NFCタグ併設 | 素材コントラストが不足する場合 |
| オンデバイス生成が重い | p5.js + Tone.js | 静止画生成、音数削減、事前生成プリセット | 低スペック端末で操作不能になる場合 |

## 11. 未決事項

- 第三者ファクトチェック未実施のため、仕様確定前に別セッションでfact-checkする
- 認識対象を何種類にするか。M1は1種、M2以降は3〜5種を推奨するが、実物の候補は未確定
- 対象物の材質、サイズ、表面状態、NFCタグの貼付・埋め込み可否が未確定
- レーザー彫刻の対象素材と加工条件が未確定。QR/画像ターゲットの読み取り試作が必要
- 持ち帰りをNFC主導にするか、QR主導にするか、併設にするかはユーザー確認が必要
- 実機検証で限定公開テストURLまたは一時トンネルを使うか、同一LAN内HTTPSに限定するかはユーザー承認が必要
- ARをM3まで含めるか、M2までを検証デモの完了範囲にするかはユーザー確認が必要
- 生成作品を「抽象ビジュアル+音」に限定するか、撮影画像のスタイル変換まで含めるかは未確定
- PWAオフライン対応をM2必須にするか、M2.5以降に回すかは通信環境の確認後に決める
- アプリ名 `camera-ar-gomi-demo` は仮称。実装前に名称確認が必要

## 12. 自律判断ログ

- 正式なアイデアシートがないため、`docs/decisions/2026-07-06_技術ベース確定.md` の D2「カメラ+AIゴミ判別」と D3「偶然性×規則性のジェネレーティブアート」、およびユーザー直接依頼を上流として扱った
