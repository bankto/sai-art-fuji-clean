## 1. 概要

- 目的: 藤クリーン リサイクルセンターの屋外施設で、スマホGPSの緯度経度を自前の施設図上へ射影し、現在地と現在ゾーンを表示できるか検証する。
- 位置づけ: 検証用デモ2件目。1件目の `camera-ar-gomi-demo` とは別アプリとする。
- ターゲットユーザー:
  - 体験者: 施設を訪れるデモ参加者・関係者。
  - 検証担当者: 現地キャリブレーションとスマホ実機確認を行う運営・開発担当者。
- 成功の定義:
  - スマホブラウザで施設図が表示され、位置情報の許可操作ができる。
  - 取得した緯度経度が画像座標へ変換され、現在地マーカーが施設図上に表示される。
  - マーカー位置に対応するゾーン名、または施設範囲外であることがテキストで表示される。
  - 権限拒否・取得失敗・施設範囲外・開発時に、明示操作でデモ用座標へ切り替えて一連の表示を確認できる。
  - 位置情報をサーバーへ送信せず、端末内だけで処理する。

## 2. スコープ

### やること(In)

- `docs/assets/fuji-clean-facility-map.png` を原図とする施設図表示。
- Geolocation APIによる現在位置の取得、取得状態と精度の表示。
- キャリブレーションデータを使った緯度経度から画像座標への射影。
- 施設図上の現在地マーカー表示。
- 簡易ポリゴンまたは矩形によるゾーン判定とゾーン名表示。
- 権限拒否、取得失敗、精度不足、施設範囲外の案内。
- デモ用フォールバック座標への明示的な切り替え。
- GitHub Pagesのサブパス `/sai-art-fuji-clean/local-map-gps-demo/` を想定した静的SPAの構成。

### やらないこと(Out)

- Mapbox、Google Mapsその他の地図API。
- 本番ツアーの予約、受付、参加者管理、運用管理。
- ユーザー認証、課金、多人数同期、位置共有。
- 屋内測位、GPS以外のビーコン測位。
- ナビゲーション、ルート検索、目的地案内。
- 位置履歴の保存、分析、サーバー送信。
- バックエンド、データベース、外部API連携。
- 初回実装時点での現地キャリブレーションと実機精度保証。
- ユーザー承認前のpush、初回公開、配信設定の確定。

## 4. 機能仕様

### 4-1. 施設図表示

- 挙動:
  - アプリ起動時に施設図を縦横比を維持して表示する。
  - マーカーとゾーン表示は施設図と同じ重ね合わせ領域に描画し、画面サイズが変わっても位置関係を保つ。
- 入力/出力:
  - 入力: `fuji-clean-facility-map.png`、画面サイズ。
  - 出力: レスポンシブな施設図と重ね合わせレイヤー。
- エッジケース:
  - 画像読込失敗時は位置取得を開始せず、「施設図を読み込めませんでした」と再読み込み案内を表示する。

### 4-2. GPS取得

- 挙動:
  - ユーザーが「現在地を表示」を押した後に `navigator.geolocation.getCurrentPosition` で位置情報を1回取得する。
  - 取得中であることを表示し、成功時は緯度・経度・精度を端末内のメモリだけに保持する。
  - 初回デモでは継続追跡を行わず、再取得はユーザーの「現在地を更新」操作で行う。
- 入力/出力:
  - 入力: ユーザー操作、ブラウザの位置情報権限。
  - 出力: `latitude`、`longitude`、`accuracy`、取得時刻、取得状態。
- エッジケース:
  - API非対応、権限拒否、タイムアウト、位置取得不能を区別して案内する。
  - 精度が設定値を超える場合は「精度が低いため表示は目安です」と示す。マーカーは表示できる場合に表示し、精度値も併記する。
  - Geolocation APIを利用できないHTTP環境ではHTTPSまたはlocalhostで開くよう案内する。

### 4-3. 画像座標への射影とキャリブレーション

- 挙動:
  - 画像上の基準点と実緯度経度の対応表を、アプリ内の差し替え可能なデータとして持つ。
  - 基準点は最低3点の非一直線な組とし、緯度経度から画像の正規化座標 `x` / `y`（各0〜1）への2次元アフィン変換係数を算出または事前生成する。
  - 基準点が3点を超える場合は、全基準点に対する残差の二乗和を最小化する最小二乗法でアフィン変換係数を近似する。
  - 初回実装は仮の基準点データで動作させ、人間が現地測定した値へデータ差し替えできる構造にする。
- 入力/出力:
  - 入力: GPS座標、`CalibrationPoint[]`。
  - 出力: 正規化画像座標、射影可否。
- エッジケース:
  - 基準点が3点未満、一直線上、重複、数値不正の場合は設定エラーとして実GPSマーカーを表示せず、デモ位置を案内する。
  - 射影結果が画像範囲外の場合は施設範囲外として扱う。

### 4-4. 現在地マーカー

- 挙動:
  - 射影結果の正規化座標に、施設図と判別できる現在地マーカーを表示する。
  - マーカー近傍または状態欄に「現在地」と精度をテキスト表示する。
- 入力/出力:
  - 入力: 正規化画像座標、GPS精度、位置取得状態。
  - 出力: 現在地マーカー、状態テキスト。
- エッジケース:
  - 施設範囲外では図の端へ丸めず、実GPSマーカーを非表示にして「施設範囲外」と表示する。
  - デモ用座標は実GPSと見分けられる「デモ位置」表記を必ず付ける。

### 4-5. ゾーン判定

- 挙動:
  - ゾーンを正規化画像座標の矩形またはポリゴンと名称で定義する。
  - 現在地が含まれるゾーンを判定し、ゾーン名をテキスト表示する。
  - 初回実装のゾーン境界と名称は、施設図を基にした仮データとして分離し、後から差し替え可能にする。
- 入力/出力:
  - 入力: 現在地の正規化画像座標、`ZoneDefinition[]`。
  - 出力: ゾーンID、ゾーン名、判定状態。
- エッジケース:
  - 複数ゾーンに含まれる場合はデータ定義順で先のゾーンを採用する。
  - 画像内だがどのゾーンにも含まれない場合は「ゾーン外」と表示する。
  - 境界上はゾーン内として扱う。

### 4-6. デモ用フォールバック座標

- 挙動:
  - 「デモ位置を表示」ボタンは、位置取得状態によらず開発・検証用の恒常導線として常時表示する。
  - API非対応・権限拒否・取得失敗・施設範囲外のときは、デモ位置で確認できる旨を案内し、このボタンを視覚的に強調する。
  - ボタン押下後、あらかじめ定義した正規化画像座標を使ってマーカーとゾーン判定を表示する。
  - 実GPS取得後も、検証目的でデモ位置へ切り替えられる。
- 入力/出力:
  - 入力: ユーザー操作、`DemoPosition[]`。
  - 出力: デモ位置マーカー、ゾーン名、「デモ位置」表示。
- エッジケース:
  - フォールバックは位置情報権限を暗黙に再要求しない。
  - デモ位置データが不正な場合はマーカーを表示せず設定エラーを示す。

## 5. 非機能要件

| 分類 | 目標値・受入条件 | 検証方法 |
|---|---|---|
| 対応環境 | 最新のiPhone SafariとAndroid Chromeを優先し、PC Chromeのlocalhostでもデモ位置を確認できる | 各対象環境で起動し、施設図表示、位置取得またはデモ位置、ゾーン表示を操作確認する |
| 性能・容量 | 施設図読込後、位置取得成功またはデモ位置選択から1秒以内を目安にマーカーとゾーンを更新する | ブラウザ開発者ツールと実機操作で更新待ちが継続しないことを確認する |
| 可用性・復旧 | API非対応・拒否・タイムアウト・取得不能・画像読込失敗を画面で識別でき、再試行またはデモ位置への導線がある | 各状態をmockまたはブラウザ設定で再現し、案内と復旧操作を確認する |
| アクセシビリティ | 状態を色だけに依存せずテキストでも示し、主要操作はキーボード操作と十分なタップ領域を持つ | キーボード操作、画面拡大、目視で確認する |
| セキュリティ・プライバシー | 位置情報をサーバー送信・永続保存せず、端末メモリ内だけで処理する。権限要求前に用途を示す | Networkタブで位置データを含む送信がないこと、Storageに位置履歴がないことを確認する |
| ログ・監視 | 外部監視・解析は設けない。開発ログに緯度経度を常時出力しない | ソース確認とブラウザコンソール確認を行う |

## 6. 技術構成

- 種別: Webアプリ（静的SPA）
- 想定アプリ名・配置: `local-map-gps-demo` / `apps/local-map-gps-demo/`
- 技術スタック:
  - Vite + TypeScript + HTML/CSS。
  - ブラウザ標準のGeolocation API。
  - 施設図とHTML/SVGまたはCanvasの重ね合わせレイヤー。
  - GitHub Pagesの既存ワークフローにサブパスを追加する構成。
- 選定理由:
  - 1件目と同じ静的SPA構成に揃えられ、サーバー、APIキー、課金なしで最小デモを構築できる。
  - 自前画像上の座標変換とゾーン判定はTypeScript内で完結し、外部地図APIを必要としない。
  - キャリブレーション点とゾーンをデータとして分離し、現地測定後の差し替え範囲を限定できる。
- ランタイム・パッケージマネージャーのバージョン固定方法:
  - 既存の `camera-ar-gomi-demo` と運用を揃え、CIの `actions/setup-node` で `node-version: 24` を指定する。新アプリへの `.nvmrc` と `package.json` の `packageManager` フィールド追加は必須としない。
  - `package-lock.json`をコミット対象とし、CIはNode.js 24環境で `npm ci` を使う。
- GitHub Pagesワークフローの変更:
  - `.github/workflows/pages.yml` の `paths:` に `apps/local-map-gps-demo/**` を追加し、新アプリの依存関係インストール・buildと、生成物を `public/local-map-gps-demo/` へコピーする処理を追加する。
- 環境と差分:
  - local: Vite開発サーバー。PCはlocalhostでデモ位置、対応端末はHTTPS環境で実GPSを確認する。
  - test: `npm run lint`、`npm run test`、`npm run build`。位置情報と射影はmock座標を使う。
  - staging: 該当なし。必要になった場合はユーザー承認後に定義する。
  - production: GitHub Pagesの `/sai-art-fuji-clean/local-map-gps-demo/`。初回公開はユーザー承認後に行う。

## 7. データ(Webアプリの場合)

### データモデル

| 名称 | 主な項目 | 保存先・制約 |
|---|---|---|
| `CalibrationPoint` | `id`, `latitude`, `longitude`, `imageX`, `imageY` | TypeScriptまたはJSONの静的データ。`imageX/Y`は0〜1、最低3点、ID重複不可 |
| `ZoneDefinition` | `id`, `name`, `shape`, `points` | 静的データ。座標は0〜1、ID重複不可、表示順を判定優先順とする |
| `PositionState` | 緯度、経度、精度、取得時刻、状態 | 端末メモリのみ。永続保存・送信しない |
| `DemoPosition` | `id`, `label`, `imageX`, `imageY` | 静的データ。実GPSではないことが分かる名称を必須とする |

- マイグレーション: DBを使わないため該当なし。データ構造変更時は型定義とfixtureを同時に更新する。
- 初期データ: 仮の基準点、仮ゾーン、デモ位置をアプリに同梱する。仮データであることをコメントとREADMEに明記する。
- ロールバック: キャリブレーションまたはゾーンデータの直前版へGitで戻せる単位に分離する。履歴改変は行わない。
- バックアップ・保持期間・削除手順: ユーザーデータを保存しないため該当なし。

## 8. API・外部連携(Webアプリの場合)

- 外部HTTP API: 該当なし。
- ブラウザAPI: Geolocation APIのみ使用する。
  - 入力: ユーザー操作と位置情報権限。
  - 出力: WGS84の緯度・経度、精度、取得時刻、またはエラーコード。
  - 認証: 該当なし。ブラウザの位置情報権限を使用する。
  - タイムアウト: 実装時に有限値を設定し、タイムアウト後は再試行とデモ位置を案内する。
  - リトライ: 自動連続リトライはせず、ユーザー操作で再取得する。
- 外部サービス障害、レート制限、冪等性: 外部サービスを使わないため該当なし。
- 環境変数: 該当なし。APIキーや秘密情報を持たない。

## 9. 認証・権限・エラー設計(Webアプリの場合)

- ユーザー認証・ロール: 該当なし。
- 位置情報権限:
  - 要求前に「現在地を施設図上に表示するために使用し、サーバーへ送信しない」と表示する。
  - 許可時は位置取得を実行する。
  - 拒否時はブラウザ設定での再許可案内とデモ位置ボタンを表示し、再要求を繰り返さない。
- エラー表示:
  - API非対応、権限拒否、タイムアウト、取得不能、精度不足、施設範囲外、設定不正、画像読込失敗を識別可能な日本語で表示する。
  - 内部例外や座標値そのものを画面へ露出しない。
- セッション、CSRF: サーバー・認証・状態変更APIがないため該当なし。
- 入力検証・出力エスケープ: 静的なキャリブレーション・ゾーンデータを起動時に検証し、名称表示はHTML文字列として直接挿入しない。

## 10. テスト戦略

| 対象 | テスト種別(unit / integration / E2E / security) | 主要ケース | 実行コマンド |
|---|---|---|---|
| 射影計算 | unit | 基準点一致、画像中央、画像外、基準点不足・重複・一直線 | `npm run test` |
| ゾーン判定 | unit | 矩形/ポリゴン内、境界上、複数重複、ゾーン外 | `npm run test` |
| 位置取得状態 | integration | 成功、拒否、タイムアウト、非対応、精度不足、施設範囲外、デモ切替 | `npm run test` |
| 画面とサブパス | E2E相当の手動確認 | Pages想定パスで施設図・マーカー・ゾーン・案内が表示される | `npm run build`後にプレビューし、ブラウザで操作確認 |
| プライバシー | security / 手動確認 | 位置情報の外部送信・永続保存・コンソール出力がない | Network / Storage / Consoleを確認 |
| 型・ビルド | integration | TypeScriptエラーなし、静的アセット解決、Pages用base反映 | `npm run lint` / `npm run build` |

- 既存のリポジトリ検証を壊さないことを確認する。新アプリでは射影・ゾーン判定・状態遷移のunit testを追加する。
- 外部サービスは使わない。Geolocation APIはtestでmockし、実接続相当の確認は現地スマホ実機で人間が行う。

## 11. 運用・リリース

- ログ・メトリクス・アラート:
  - 外部ログ、アクセス解析、位置情報監視は設けない。
  - ソフトウェア検証は実装担当がlint / test / build結果を作業報告へ記録し、現地検証は人間が端末・ブラウザ・測定地点・精度を記録する。
- デプロイ:
  - 既存のGitHub Pagesワークフローへ `local-map-gps-demo` のbuildとサブパス配置を追加する想定とする。
  - pushおよび初回公開は、オーケストレーターがユーザー承認後に実行・確定する。
- ロールバック:
  - DB変更はない。直前の正常な静的成果物へ戻し、必要に応じてキャリブレーションデータだけを直前版へ戻す。
- 障害時の切り分け:
  - 施設図読込、HTTPS・権限、GPS取得、キャリブレーション、ゾーン判定、Pagesのbase pathの順に確認する。
  - データ復旧は該当なし。問い合わせ導線は本番ツアー運用外のため設けず、検証担当者が結果を作業記録へ残す。

- 実機精度確認: 藤クリーン リサイクルセンター現地で、iPhone SafariとAndroid Chromeの精度、安定性、施設範囲外判定を確認する必要がある。受容できる精度閾値は実測後に確定する。
- 仮ゾーンの正式な名称、境界、重複時の業務上の優先順位は未決。初回実装ではデータ差し替え可能な仮定義を使う。

## 14. 自律判断ログ

- GPSは最小デモとして1回取得方式を採用し、継続追跡はOutとした。位置履歴を持たず、実装とプライバシーの範囲を小さくするため。後から継続追跡へ変える場合は状態管理と停止処理の追加が必要。
- 射影方式は最低3つの非一直線な基準点を使う2次元アフィン変換とした。自前施設図への最小実装として扱いやすく、基準点データを現地測定後に差し替えられるため。実測誤差が大きい場合は変換方式の再検討が必要。
- ゾーンは矩形とポリゴンの両方を表現できる静的データとし、重複時は定義順を優先すると仮決定した。初回実装を単純にしつつ後から境界を修正しやすくするため。
- 権限拒否・取得失敗・施設範囲外では自動的にデモ位置へ切り替えず、ユーザーの明示操作を必須とした。実GPS表示との誤認を防ぐため。
- デモ位置ボタンは状態によらず常時表示すると決定した。実GPSの成功後を含め、検証担当者が任意の状態からデモ表示へ切り替えられる利便性を優先するため。
- Node/npmの固定方式は既存の `camera-ar-gomi-demo` とCI運用を揃え、`node-version: 24` を採用し、新アプリだけに `.nvmrc` / `packageManager` を必須化しない。リポジトリ内で固定方式が分かれて保守判断が増えることを避けるため。
