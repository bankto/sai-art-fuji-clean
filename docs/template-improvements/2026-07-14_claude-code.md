## 2. この案件から見た観察

- **レビュー担当(Claude Code)としての立場**: `docs/reviews/` に5件のレビュー(仕様2ラウンド・実装1ラウンド分含む)を書く一連の流れは、`## 判定` 直下1行の機械読みルールのおかげで Cursor 側の判定ロジックが単純化されており、r1→反映→r2で問題なく収束していた。レビュー対象がハードウェア依存機能(GPS・カメラ)のとき、「ブラウザで再現できない」ことの扱いに毎回同じ迷いが出た(後述 Should)
- **調査・専門サブエージェントとしての立場**: `reference-hunter` / `niche-spot-scout` / `official-source-researcher` / `fact-checker` の分業は Phase 1(リサーチ)で機能したが、これは dev/website テンプレ双方に存在しない「独自追加フェーズ」であるため、テンプレの改定(sync_shared.py 対象)には乗らず、逆輸入(手動反映)に頼っていた
- **成果物の置き場所としての立場**: 人間がフィールドで行う物理作業(現地GPS計測・現物撮影→Teachable Machine学習)が2件続き、そのための非技術者向け手順書(`docs/decisions/2026-07-13_teachable-machine-shooting-checklist.md`)を即興で作る必要があった。テンプレの「人間が確認・意思決定するポイント」表は基本的に成果物の承認(はい/いいえの判断)を想定しており、「非技術者が数日がかりで行う物理作業の手順書」という成果物種別は無かった
- **複数プロダクト並走としての立場**: `apps/camera-ar-gomi-demo` と `apps/local-map-gps-demo` が同一 GitHub Pages に同居し、`.github/workflows/pages.yml` は2アプリ分のビルド・コピー処理を手作業(Codexへの実装委譲)で重複記述している。テンプレの `apps.mdc` / `implement.md` は「1プロダクト=1フォルダ」までしか定義しておらず、複数プロダクトを同一デプロイ先に同居させる際の配線(デプロイ設定への登録漏れ)はレビュー観点にも入っていない

## 3. テンプレートへの改善提案

### Must

**M1. テンプレを部分カスタマイズ・フェーズ拡張したプロジェクトへの改定反映手順が無い**

- 対象テンプレ: 両方(README/CHANGELOG運用)
- 問題: `sync_shared.py` はテンプレ間(ai-dev-workflow ⇔ ai-website-workflow)の共通ファイル同期のみが対象。本案件のように「テンプレを土台にフェーズを5つに拡張し、AGENTS.md・orchestrate.md・各種 `.mdc` を大幅に書き換えた」プロジェクトは同期対象外になり、テンプレ側の改善(2026-07-07〜07-12 の一連の改定)を都度読み比べて手動で「逆輸入」する専用ドキュメント(`docs/decisions/2026-07-06_テンプレ改善の逆輸入.md` 等)を書く必要があった。実績としてこの逆輸入は既に2回発生している
- 提案: テンプレ側 README または CHANGELOG に「フェーズ構成をカスタマイズしたプロジェクトが、CHANGELOG の各エントリを自分のプロジェクトへ手動で当てる」ための簡易チェックリスト(例: 「orchestrate.md の該当節だけ diff で見る」「AGENTS.md の対応する行を探す」)を追加する。テンプレ本体の改修ではなく、逆輸入作業を毎回スムーズにするためのガイドで足りる
- 根拠: `docs/decisions/2026-07-06_テンプレ改善の逆輸入.md`、`docs/decisions/2026-07-05_全フェーズオーケストレーション改定.md`(いずれも手動反映の記録)
- 想定コスト: 小(README/CHANGELOGへの追記のみ)

**M2. 複数プロダクトを同一デプロイ先に同居させる際の配線がテンプレでカバーされていない**

- 対象テンプレ: 両方(dev/website 共通。`apps.mdc` / `implement.md` / `implement-review.md`)
- 問題: `apps.mdc` は「1プロダクト=`apps/` 直下の1フォルダ」までしか定義していない。本案件では2つ目のデモを追加した際、`.github/workflows/pages.yml` に新しい `working-directory` ブロック・`paths` トリガー・コピー先を手作業で追記する必要があったが、これを Codex への実装委譲時にリマインドする仕組みも、`implement-review.md` のチェック項目もない。「実装は完了したがデプロイ設定に登録し忘れる」事故が起きやすい構造
- 提案: `apps.mdc` に「2つ目以降のプロダクトを追加する場合、共有デプロイ設定(GitHub Actions 等)への登録が必要」の一文を追加。`implement-review.md` のレビュー観点に「新規・変更したプロダクトがデプロイ設定(ワークフロー/ビルド対象)に反映されているか」を1項目追加する
- 根拠: `.github/workflows/pages.yml`(2アプリ分のビルド・コピー処理が個別に重複記述されている実装)
- 想定コスト: 小(定型文の追加のみ)

### Should

**S1. 非技術者向けの物理作業手順書という成果物パターンがテンプレに無い**

- 対象テンプレ: 両方
- 問題: 「人間が確認・意思決定するポイント」の枠組みは基本的に成果物の承認(Go/No-Go判断)を想定しており、「専門知識のない人が現場で数日がかりの物理作業(現地測量・現物撮影)を行うための手順書」という成果物種別が定義されていない。本案件では即興で `docs/decisions/` に手順書(専門用語「クラス」を「種類」に言い換える等の配慮込み)を作成したが、置き場所も雛形もなかった
- 提案: `docs/workflows/implement.md`(または `orchestrate.md`)に「実装完了後に人間の現地/物理作業が必要な場合は、非技術者にも読める手順書を作成し、置き場所(例: `docs/handoffs/` 新設 or 既存 `docs/decisions/`)を明示する」旨の指針を追加する
- 根拠: `docs/decisions/2026-07-13_teachable-machine-shooting-checklist.md`
- 想定コスト: 小(指針1文+置き場所の決め)

**S2. デバイス依存機能(GPS・カメラ等)のレビューで「未検証」の扱いが曖昧**

- 対象テンプレ: 両方(`implement-review.md`)
- 問題: `implement-review.md` は「画面操作が関係する場合はブラウザ(Playwright MCP等)でも確認する」とあるが、位置情報・カメラのように委譲先(非対話・実機を持たない環境)ではブラウザ確認自体が困難な機能への具体的な代替検証方法(モック座標・フォールバック値でのユニットテスト等)の指針がない。本案件の2本のデモ(GPS/カメラ)で毎回レビュー担当が「実機確認は未検証」と書くだけで済ませる運用になっていた
- 提案: `implement-review.md` に「ブラウザ/委譲先で再現できないデバイス機能は、モック値・フォールバック値を用いた検証方法を仕様書に明記させ、レビューではそのモック経路が動くことを確認する」旨を追記する
- 根拠: `docs/reviews/2026-07-13_local-map-gps-demo_m1-m3_implement-review_r1.md`、`docs/reviews/2026-07-10_camera-ar-gomi-demo_implement_review.md`
- 想定コスト: 小

**S3. 複数プロダクトが並走する案件での `docs/status.md` の書式指針が薄い**

- 対象テンプレ: 両方(`docs/status.md` テンプレート)
- 問題: テンプレの `docs/status.md` はフェーズ進捗表・直近のアクションとも単一プロダクトを前提にした書式で、複数プロダクトが別々の状態(1件目=出張待ち、2件目=現地キャリブ待ち)で並走する場合の書き方が決まっていない。本案件では「フェーズ進捗」表のメモ欄・「直近のアクション」に無理やり2系統を詰め込んでいる
- 提案: `docs/status.md` テンプレ、または `orchestrate.md`「docs/status.md の運用」に「複数プロダクトが並走する場合はメモ欄にプロダクト名を先頭に付けて併記する」程度の軽い書式ルールを追記する(表の分割までは不要、肥大化しない範囲で)
- 根拠: `docs/status.md` の「フェーズ進捗」「直近のアクション」節(1件目/2件目が同じ行・欄に混在)
- 想定コスト: 小

**S4. 人間作業待ちが長期化したときの「催促・確認頻度」の目安がない**

- 対象テンプレ: 両方(`orchestrate.md`「セッションの切り替え」「区切りごとの最終応答」)
- 問題: `再開情報`/`ブロッカー` 欄は「今何を待っているか」を書く欄はあるが、待ちが数日〜1週間規模になったときにオーケストレーターがどの頻度で状況を確認する・ユーザーに催促するべきかの目安がない。本案件は出張待ち・現地キャリブ待ちが複数日続いており、放置と過剰確認の中間basisが無いまま運用していた
- 提案: 「区切りごとの最終応答」に、ブロック中が一定期間(例: 目安1週間)続いた場合は次回応答で軽く状況確認を促す、程度の目安を追記する
- 根拠: `docs/status.md`「再開情報」のブロッカーが複数日継続(2026-07-13時点で現地キャリブ・出張撮影の両方待ち)
- 想定コスト: 小

### Could

- 対象テンプレ: 両方(将来検討)

- 提案: 今回は必須ではないが、次にテンプレへリサーチ的工程を足す機会があれば、本案件の `.claude/agents/` 構成を出典として参照する
- 根拠: 本案件 Phase 1 の運用実績(`docs/research/2026-07-02_廃棄物クリエイティブ事例/`)
- 想定コスト: 中(サブエージェント定義+ワークフロー新設が必要になるため次回検討)

## 4. 既存テンプレに既に入っていて、本案件でも効いたもの(維持推奨)

- **レビュー判定の機械読み**(`## 判定` 直下1行): 仕様レビュー r1→r2、実装レビュー r1 のいずれも Cursor 側が本文を読み返さず判定行だけで次工程へ進めており、往復コストを下げていた
- **非対話・ハング対策**(Windows: `cmd /c "... < nul"` + 長文は `_prompt.md` 分離): 本案件の委譲(Codex/Claude Code とも)でハングによる STOP は発生していない
- **モデル・effort の毎回明示ルール**(2026-07-10 導入): 本案件では軽量修正(camera-ar-gomi-demo の UI整理等)に `medium`、仕様レビュー等に `high` を使い分けており、既定任せによるトークン浪費は再発していない
- **実行確認モデルの取得フロー**(`--output-format json` の `modelUsage`): `docs/status.md`「直近のAI実行ログ」で指定値と確認値が分けて記録され続けており、機能している
- **`docs/status.md` の肥大防止(履歴系2節の20行上限+`docs/history/` 退避)**: 本案件は2026-07-13時点で決定事項ログ・AI実行ログとも20行未満に収まっており、退避ルール導入前に起きていた肥大は再発していない
- **作業報告(`docs/reports/`)の存在チェックとテンプレ**: 実装のたびに `mN` 単位の報告が作成されており、レビュー担当が「報告の記載だけで判断せず対象ファイルを直接確認する」動きと噛み合っていた

## 5. 未決事項・自律判断ログ

- 実行確認AI/モデル/effort は非対話実行のため自己確認できず、指示どおり「不明」と記載した(オーケストレーターが `modelUsage` から埋める前提)
- 提案の優先順位(Must/Should/Could の割り振り)は、テンプレ本体の恒久的な欠陥かどうかで判断した: 「複数プロダクト同居」「テンプレ拡張プロジェクトへの改定反映」は次のプロジェクトでも確実に再発するため Must、それ以外は本案件固有の摩擦だが再発可能性があるため Should とした。この重み付けはレビュアー(Claude Code)の裁量による自律判断であり、最終的な採否はユーザー判断に委ねる
- M1・M2 はテンプレ本体(`D:\workspace\ai-templates`)の改修を伴うが、本指示(前提③)により本ドキュメントでは変更を行っていない。採用する場合は改めてテンプレ側への反映作業(Codex 等への別委譲)が必要
