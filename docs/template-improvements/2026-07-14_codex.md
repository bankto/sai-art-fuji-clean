## 対象と前提

横断テンプレート正本 `D:\workspace\ai-templates`（版 `2026.07.12.2`）と、本案件のドキュメント、CSV、レビュー反映、2アプリ + research-site の Pages 同居構成を照合した。テンプレート本体は変更せず、作成担当が再現性高く完走するための改善案だけを記録する。

## 作成担当として詰まった点

### ドキュメント作成

- 本案件の5フェーズは既存テンプレートに無く、雛形、上流リンク、ファイル名、status の書き方を案件側で拡張する必要があった。
- 旧絶対パスが正本移転後も文書に残り、作成時に「記載されたパス」と「実在する正本」のどちらを採用するか確認が必要になった。
- 人間が現地で行う撮影・GPS測定の手順書は既存の成果物種別に当てはまらず、`docs/decisions/` に置く判断を案件側で行った。

### 実装

- 2つの Vite アプリと静的 `research-site` を1つの Pages artifact に束ねるため、アプリ追加時に workflow の paths、cache dependency、build、copy、公開先、各アプリの base path を一緒に更新する必要があった。
- GPS・カメラなど実機依存機能は、Codex 側の lint / test / build だけでは完了を証明できない。モック経路と人間の実機作業を分けて記録する型が不足していた。
- 実機フィードバックで導線を一時非表示にする場合、削除・CSS非表示・設定フラグのどれを選ぶかと、復帰条件を仕様から読み取れないことがある。

### レビュー指摘の反映

- `## 判定` の機械読みと Must 優先は明確で、反映作業の入口として非常に有効だった。
- 一方、Should / Could の採否、レビュー本文のチェック更新、仕様変更を伴う指摘の差し戻し先は複数ファイルにまたがる。反映対象表があると、コードだけ直して仕様・報告・レビューの追跡を漏らす事故を減らせる。
- デバイス依存項目は「未検証」のまま承認される場合があり、Codex が何を自動検証し、何を人間へ渡すかを作業報告に揃えて書く型が必要だった。

### CSV・データ整形

- 本格リサーチから採用事例CSVとニッチ照合CSVを作成したが、列定義、出典URL、検証状態、重複キー、文字コード、改行の扱いをテンプレートが定義していない。
- 調査担当から作成担当へ渡す際、raw source と採用済み行の境界が案件固有になりやすい。CSVごとのデータ辞書と検証状態列があれば、整形とファクトチェックを分離しやすい。

### Pages 同居

- `.github/workflows/pages.yml` は3成果物を1つの `public/` に集約する。新しいアプリを追加するたびに複数箇所の手動更新が必要で、実装完了と公開配線完了が別状態になりうる。
- ルートサイトとサブパスアプリでURL・アセットパス・404時挙動が異なるため、「各アプリの build 成功」だけでは共有 Pages の検証として不足する。

## 効いた既存ルール

- `## 判定` 直下1行の機械読みと、条件付き承認 → Codex反映 → 再レビューの分業。
- Codex は対象を修正し、Claude Code はレポートのみを書く役割境界。
- `_prompt.md` と Windows の stdin close による非対話ハング対策。
- マイルストーン単位の作業報告、lint・既存テスト全件・build の検証。
- 実行指定と実行確認の分離、モデル・effort の毎回明示。
- ASCII slug の新規ファイル名規則。日本語本文を保ちつつリンク事故を減らせる。
- `2026.07.12.2` の status 短縮ルール。現在の課題はルール不足ではなく、点検の機械補助である。

## 未決事項・自律判断ログ

- 第3バリアントは必要性がある一方、既存2テンプレートとの共通ルール重複が大きい。そのため即時新設を Must にせず、任意モジュールまたはプリセットの試行を Could とした。
- `docs/handoffs/` 新設は成果物種別を増やすため、まず `docs/setup.md` 拡張との比較が必要。統合ノートでは人間承認事項として残す。
- CSV提案は本案件の実績に基づくが、現行 dev / website の全案件に必須ではない。リサーチモジュール採用時だけ有効にする前提で Should とした。
- 実行確認モデルと effort は構造化ログを参照できないため不明とした。指定値を確認値へ転記していない。
