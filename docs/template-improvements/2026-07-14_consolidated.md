## 新バリアント候補（research → ideate → multi-demo）の要否

### 結論

候補としては必要。ただし、次版で直ちに3つ目の完全コピーを作るのではなく、`ai-dev-workflow` に追加できる任意モジュールまたは生成プリセットとして1案件試行するのが妥当である。

### 必要と判断する理由

- 本案件で5フェーズが実際に完走し、リサーチ、まとめサイト、アイデア選定、2つの検証デモという再利用可能な流れが確認できた。
- 収集担当と検証担当のセッション分離、CSV整形、上流リンク、multi-demo の比較は既存2テンプレートの範囲外である。
- 案件側の独自拡張とテンプレ改善の逆輸入が複数回発生しており、偶発的な1回限りの課題ではない。

### すぐ独立バリアントにしない理由

- `AGENTS.md`、orchestrate、通知、Git、モデル・effort、レビュー規則の大部分は既存テンプレートと共通で、完全コピーは同期対象を3系統へ増やす。
- まとめサイトが必須でないリサーチ案件や、デモが1本だけの案件もあり、5フェーズ固定は過剰になりうる。

### 推奨する判定ゲート

1. 共通基盤は既存 `ai-dev-workflow` を使う。
2. research、research-site、ideate、multi-demo/status を任意モジュールとして切り出す。
3. 次の類似案件で導入し、フェーズ削除の容易さ、doctor、同期負荷を測る。
4. 2件以上で同じ組み合わせが再利用されたら、独立バリアント化を人間が承認する。

## テンプレート本体への反映手順案

今回は実行しない。人間承認後、次の順で別作業として進める。

1. **採否決定**: TI-01〜TI-05 と、第3バリアントを「任意モジュールで試行」とする方針を人間が承認する。
2. **変更単位を分割**: まずパス修正・commit優先順位・status型・複数デプロイ観点を小さな版にまとめ、ラッパーやモジュールは別版にする。

4. **移行情報更新**: `CHANGELOG.md`、`MIGRATIONS.md`、`TEMPLATE_VERSION`、各 `.template-version` を同じ変更単位で更新する。
5. **機械検証**: `python3 scripts/sync_shared.py --check` と `python3 scripts/template_doctor.py` を実行し、旧パスが0件であることも検索する。

8. **レビュー**: Claude Code がテンプレート整合レビューを行い、判定に従って Codex が反映、必要なら再レビューする。
9. **人間承認と記録**: 新バリアントの独立化は試行結果を提示して別途承認を得る。commit / push はテンプレート側の適用ルールに従う。
