## 対象と位置づけ

Cursor から共有された12件の観察を、内容を落とさず「観察」「影響」「改善方向」に整形した。対象は横断テンプレート正本 `D:\workspace\ai-templates` の版 `2026.07.12.2`。今回は観察の記録のみで、テンプレート本体は変更していない。

## 観察事項

### 1. 第3のテンプレートバリアントがない

- 観察: 本案件は「リサーチ → まとめサイト → アイデア → 仕様 → 実装」の5フェーズで、さらに複数の検証デモを並走した。現行テンプレートは `ai-dev-workflow`（要件 → 仕様 → 実装）と `ai-website-workflow`（要件 → デザイン台帳 → 仕様 → 実装）の2種で、リサーチ主導・複数デモ検証型を直接扱えない。
- 影響: 本家のルールを案件側で拡張し、後日テンプレート改善を手動で逆輸入する作業が繰り返し発生した。
- 改善方向: `research → ideate → multi-demo` を想定した第3バリアント、または既存テンプレートへ追加できるフェーズモジュールを検討する。

### 2. ドキュメント内の絶対パスが陳腐化している

- 観察: 案件の `AGENTS.md` とテンプレート2種の `README.md` に `D:\workspace\_templates\...` が残る一方、現在の正本は `D:\workspace\ai-templates\` にある。
- 影響: コピー先にも古い参照が残り、セットアップや通知の正本を誤認しやすい。
- 改善方向: 正本パスを一括修正し、可能な箇所はリポジトリ相対パスまたは置換トークンにする。doctor に旧パス検出を追加する。

### 3. 人間作業ブロッカーの待ち状態を表しにくい

- 観察: 現地GPS測定や Teachable Machine 用撮影など、AIでは完了できず長期間待つ作業が複数発生した。`docs/status.md` に再開情報はあるが、「ブロッカー種別（人間 / 外部 / AI）」「待ち中に並走可能なタスク」「再開トリガー」の型が薄い。
- 影響: 待ち状態と停止状態が混ざり、別セッションが何を進めてよいか判断しにくい。
- 改善方向: 再開情報に種別・担当・並走可能タスク・再開トリガー・手順書リンクを持たせる。

### 4. 複数プロダクトと GitHub Pages の同居が複雑

- 観察: 1リポジトリにデモ2本と `research-site` があり、単一 Pages 配下へ配置している。各アプリの build、copy、base path、paths trigger、レビュー単位が単一プロダクト前提より複雑になった。
- 影響: 2本目以降のアプリで、デプロイ対象への登録漏れ、URL衝突、base path 不整合、片方だけの検証漏れが起きやすい。
- 改善方向: 複数プロダクト用の配線表とデプロイ・レビューのチェック項目を追加する。

### 5. マルチトラック進行を表しにくい

- 観察: 1件目が人間作業待ちの間に2件目へ着手するなど、複数トラックが別状態で進んだ。`docs/status.md` の「現在のタスク」は単数形である。
- 影響: 主トラック、待機トラック、次に再開すべきトラックを見失いやすい。
- 改善方向: status を肥大させない範囲で、トラックID・状態・次アクション・ブロッカー・再開トリガーの短い表を持てるようにする。

### 6. レビューから自律反映までのループは有効だった

- 観察: `## 判定` 直下1行の機械読みと、「条件付き承認 → Codex が Must を反映 → Claude Code が再レビュー」の流れが実用的に機能した。
- 影響: オーケストレーターの判定処理が単純になり、役割分担を崩さずに短い反復で収束できた。
- 改善方向: 現行ルールを維持し、判定値の lint、対象・ラウンド命名、Must 対応の追跡を強化する。

### 7. status 肥大対策は入ったが運用依存である

- 観察: 版 `2026.07.12.2` で履歴2節の20行上限、`docs/history/` 退避、全体120行目安が導入済み。一方、退避実行はオーケストレーターの注意に依存する。
- 影響: 更新時の点検を忘れると、ルールがあっても再び肥大する。
- 改善方向: 新規提案として同じルールを重ねず、doctor や軽量チェックで行数・履歴件数・テンプレ外見出しを検出する。

### 8. commit 方針の優先順位が曖昧になりうる

- 観察: ユーザー個別ルールの「commit は明示依頼時のみ」と、`orchestrate.md` の「レビュー承認区切りで commit 可」が併存する案件がありうる。
- 影響: 一般ルールを自動許可と読むか、個別ルールを上書き条件と読むかで担当者の判断が分かれる。
- 改善方向: 「ユーザー / 上位 AGENTS / 案件 status の明示方針がテンプレ既定より優先」の一文と、commit の既定値を status に固定する。

### 9. Windows の委譲コマンドにボイラープレートが多い

- 観察: `_prompt.md` と `cmd /c "... < nul"` の組み合わせはハング回避に効いたが、モデル・effort・sandbox・JSON出力などを毎回正しく書く必要がある。
- 影響: 引用符、stdin close、必須フラグの抜けによる失敗が起きやすく、委譲プロンプト作成コストも高い。
- 改善方向: AI種別・effort・プロンプトパスだけを受け取る薄い PowerShell / shell ラッパーを同梱し、実行コマンドを標準化する。

### 10. 判断事例をテンプレへ還元する記録形式がない

- 観察: `orchestrator-judgment` は「事例が溜まったら還元」とするが、事例の記録場所や項目が定義されていない。
- 影響: 良い判断や境界事例がチャット・decision・status に分散し、一般化の材料が残りにくい。
- 改善方向: 状況、選択肢、採用判断、根拠、結果、一般化可否を持つ小さな事例フォーマットを追加する。

### 11. 実機フィードバックによる一時的なUI切替を扱いにくい

- 観察: NFC / QR の一時非表示など、デモの実機確認で「削除ではなく一時的に見せない」方針が生じた。現行の仕様・実装ワークフローにはこの種のUIトグル方針がない。
- 影響: 仕様から消す、コードを削除する、フラグで残す、の判断が都度発生し、将来の復帰条件も失われやすい。
- 改善方向: 仕様に可視状態、切替方法、既定値、復帰条件、テスト対象を記録する軽量な feature-toggle 欄を追加する。

### 12. テンプレート取り込み後の差分監査が弱い

- 観察: 版を取り込んだ後に「案件固有カスタム」と「テンプレ正本由来の差分」を棚卸しする定型手順がない。
- 影響: 正本更新で案件固有ルールを上書きしたり、逆に必要な更新を取りこぼしたりしやすい。
- 改善方向: 取り込み前のスナップショット、正本との差分分類、保護対象、適用後 doctor、`.template-version` 更新を一連の監査チェックリストにする。

## まとめ

最も再発性が高いのは、古い絶対パス、複数トラックの再開情報、複数プロダクトのデプロイ配線、カスタム案件への更新取り込みである。第3バリアントは課題を直接解く候補だが、既存2テンプレートとの重複を増やすため、共通モジュール化とセットで判断する必要がある。

## 未決事項・自律判断ログ

- 12件の優先順位付けは統合ノート側で行い、この文書では観察内容を落とさないことを優先した。
- パス陳腐化は実ファイルを検索し、テンプレート2種の README と本案件の AGENTS.md に残存することを確認した。「AGENTS.md 等」という観察を、確認できた具体例に限定して表現した。
- commit の衝突は、テンプレート正本内だけでなくユーザー個別ルールとの合成時に起きる問題として整理した。
