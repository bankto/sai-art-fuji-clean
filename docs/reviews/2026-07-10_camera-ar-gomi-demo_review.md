## 総評(3行以内)

構成・マイルストーン分割・リサーチ結論の反映はいずれも質が高く、検証用デモの仕様書として骨格は妥当。
ただし、NFC検証のマイルストーン所属が節間で矛盾(M2/M2.5)、シード再現性の定義が完了条件と矛盾、実機検証用HTTPS配信手段が未定義のままスコープOutと衝突、の3点は確定前の修正が必須。
ユーザー未決2点(M2/M3範囲・NFC/QR主導)の判断材料は仕様書内に揃っている。

## 判定

条件付き承認

## 指摘事項

### Must(必ず対応)

- [x] **NFC検証のマイルストーン所属が節間で矛盾している(M2 か M2.5 か)**。§4-6 は「M2ではNFCタグへのURL書き込みをAndroid Chromeで検証する」、§7 推奨案も「M2でWeb NFC…を追加」とするが、§9 マイルストーン表では NFC持ち帰りは M2.5 で、M2 は URL/QR のみ。根拠: ユーザー未決事項「M2までを完了範囲にするか」の判断は M2 の中身に直結するため、この揺れは意思決定を曖昧にする。修正案: §9 の表を正とし、§4-6 と §7 の「M2」を「M2.5」に統一する(またはその逆に統一し表を直す)。
  - 対応: §9を正として、§4-6と§7のNFC追加時期をM2.5へ統一した。
- [x] **シード再現性の定義が §4-3 と M1.5 完了条件で矛盾している**。§4-3 はシード材料に「撮影時の簡易画像特徴」(撮影ごとに変わりうる値)を含むが、M1.5 完了条件は「認識結果から同一シードが生成され」、検証方法は「同じ対象物で2回生成し、同じURLシードで同じ…パラメータになる」とあり、同一対象→同一シードを要求するように読める。根拠: 上流 D3「偶然性×規則性」のどちらをどこで担保するかが未定義のままでは、完了条件が判定不能(spec-review 観点「完了条件が判定可能か」)。修正案: 「規則性=同一シード→同一出力」と「偶然性=撮影ごとにシードが変わってよい」を分離し、M1.5 完了条件を「1回の生成で得たシードURLを再読み込み(または別端末で開き)、同一の見た目・音のパラメータが復元される」に書き直す。あわせて §4-3 に「同一対象・同一条件で同一シードにするか、撮影ごとに変えるか」の設計判断を明記する。
  - 対応: §4-3に規則性と偶然性の分離を明記し、M1.5の完了条件と検証方法をシードURL再読み込み/別端末復元へ修正した。
- [x] **実機検証用の HTTPS 配信手段が未定義のまま、スコープOutの「本番公開、外部公開、デプロイ設定の確定」と衝突しうる**。根拠: §9 の全マイルストーンの検証方法と成功の定義(「別端末または後日の再生導線」)はスマホ実機のHTTPSを前提とするが、§6 は「開発確認: …スマホ実機のHTTPS」と書くのみで手段がない。特に M2.5 で NFCタグに書き込むURLが開発機のローカルIPだと「後日・自宅で再生」が成立せず、逆に恒久URLを用意するなら「外部公開」に踏み込む(STOP条件相当のユーザー承認事項)。修正案: 検証手段の候補(例: mkcert等のローカルHTTPS+同一LAN / トンネル / 限定公開のテストURL)を §6 または §7 に明記し、外部到達可能なURLを使う場合は §11 未決事項に「限定公開の可否・方式はユーザー承認が必要」と追加する。
  - 対応: §6にmkcert+同一LAN、一時トンネル、限定公開テストURLの候補を追記し、§11に限定公開方式のユーザー承認事項を追加した。

### Should(対応推奨)

- [x] **M2.5 完了条件「iPhoneとAndroidの少なくとも一方で読み取り」では、リサーチ未決事項1(iPhone実機でNFCタグURLが期待どおり開くかの確認)が未検証のまま完了になりうる**。根拠: リサーチはApple公式ページの本文が取得できず実機確認を未決として残しており、iPhoneでの読み取り可否は「NFC主導/QR主導/併設」のユーザー判断の主要材料。修正案: M2.5 の完了条件に iPhone での読み取り確認を含めるか、Androidのみで完了とする場合は「iPhone未確認である旨を成果物(検証記録)に必ず残す」条件を付す。
  - 対応: M2.5完了条件にiPhone読み取り確認を含め、Androidのみ完了時はiPhone未確認を検証記録へ残す条件を追加した。
- [x] **認識対象の種類数が節間で不整合**。§4-2「M1では、用意した対象物1〜3種」、§9 M1「1対象物認識」、§11「M1は1種、M2以降は3〜5種を推奨」。根拠: 完了条件の判定基準が節により変わる。修正案: M1=1種に統一。あわせて、分類モデルは対象物クラスだけでは「非対象」を判定できないため、「背景/非対象クラスを1つ加える(または閾値設計で代替する)」旨を §4-2 に明記する。
  - 対応: §4-2をM1=1種、M2以降=3〜5種へ統一し、背景/非対象クラスまたは閾値設計を追記した。
- [x] **技術候補表 D 行の 8th Wall の記述がリサーチと不整合**。仕様書は 8th Wall を Zappar と同列に「商用WebAR」「課金・契約・外部サービス判断が必要」とするが、リサーチは「2026-02-28にホスト型有料プラットフォーム終了、現行は無料・ログイン不要・セルフホスト可(XR Engineのライセンス要確認)」と評価している。根拠: 出典と矛盾する理由付けは、後から読む人の技術判断を誤らせる(結論「下書き段階では不採用」自体はリサーチ推奨と整合)。修正案: 8th Wall を Zappar と分け、「無料化されたが新体制直後でライセンス確認が必要なため不採用」と根拠を書き換える。
  - 対応: 8th Wall系/XR EngineとZappar等を分け、新体制直後のライセンス・保守性確認が必要なため下書き段階では不採用と書き換えた。

### Could(任意・改善提案)

- [x] §11 未決事項1「調査レポートの出典は未検証の注記があるため」は現状と合わない。現行リサーチに未検証注記はなく、参照日付きで公式情報確認済み(ただし第三者ファクトチェックは未実施)。「第三者ファクトチェック未実施のため、仕様確定前に別セッションで fact-check する」に文言修正する。
  - 対応: §11の文言を「第三者ファクトチェック未実施」に修正した。
- [x] スコープIn の「QR表示・読み取り」の「読み取り」の主体が機能仕様にない。OS標準カメラでの読み取りを前提にするなら、その旨を §4-6 に明記し、アプリ内QRリーダーはスコープ外と書く(iOS Safari 前提だと BarcodeDetector が使えないため、アプリ内読み取りを作るならJSライブラリ追加が必要になる)。
  - 対応: §4-6にQR読み取りはOS標準カメラ/標準機能を前提とし、アプリ内QRリーダーはM2では作らない旨を追記した。
- [ ] §4-3 エッジケース「旧バージョン: 旧生成ロジックを残すか、再生成差分を明示する」が二者択一のまま残っている。方針を決めて書く(推奨: generatorVersion ごとに生成ロジックを保持し、未知バージョンはエラー表示)か、§11 未決事項へ移す。
- [x] §10 の採用条件「認識成功率がデモ条件で8割未満」の測定条件(試行回数・照明・距離・対象物の置き方)が未定義。M1 の検証時に測定手順ごと記録する旨を添えると判定可能になる。あわせて §4-2 の信頼度閾値は「M1 実装時に実測で決める」と明記する。
  - 対応: §4-2のエッジケースに、M1実装時の実機誤認識を見て暫定閾値を決め検証記録に残す旨を追記した。

## 良い点

- マイルストーンが 0.5 刻みの縦切りで分割され、M1 が「カメラ+1対象物+1反応」の最小限動く構成になっている。テンプレにない「検証方法」列の追加も、1委譲で完結できる粒度の担保として有効
- リサーチの中核結論「マーカーレス物体追跡はWebでは高難度 → NFC/QR/画像ターゲットへ分解」が §5 補足で「読み替え」として明示され、上流要望と仕様の差分が追跡できる
- やらないこと(Out)が具体的で、課金・認証・汎用ML・ネイティブ・個人情報の除外が AGENTS.md の STOP 条件・検証デモの位置づけと整合している
- 各機能にエッジケース(カメラ拒否、アプリ内ブラウザ、音声開始UX、シード破損、NFC非対応時のフォールバック)が付いており、当日運用の失敗モードを先回りしている
- 非機能要件にプライバシー(カメラ映像は端末内処理・サーバー送信なし)と運用準備(撮影台・照明・タグ)まで記載がある
- 技術候補が5案比較され、推奨案の選定理由が「検証デモとしてリスクが低く戻しやすい」という目的に紐づいている。§12 自律判断ログで上流の扱いと読み替えの経緯が追える

## 補足

- 判定を「要修正」ではなく「条件付き承認」にした理由: Must 3件はいずれも節間の整合・文言の修正で対応でき、体験フロー・技術選定・マイルストーン構成という骨格の再設計は不要と判断したため
- ユーザー確認が必要な2点(仕様書 §11 にも記載あり)を再掲する。いずれも仕様確定前にユーザー判断が必要:
  1. **M2までを完了範囲にするか、M3(AR/レーザー彫刻試作)まで含めるか**。Must 1件目(NFCのM2/M2.5所属)を解消しないと「M2まで」の中身が確定しないため、先に修正してから判断するのがよい
  2. **持ち帰りをNFC主導 / QR主導 / 併設のどれにするか**。Should 1件目のとおり、iPhoneでのNFCタグ読み取り実機確認(リサーチ未決事項1)が判断材料になる。判断を M2.5 検証後に先送りする選択肢もある
- 本レビューは仕様書の内部整合と上流(技術ベース確定・リサーチ)との整合を対象とし、リサーチ内容自体の真偽検証(fact-check)は行っていない。仕様書 §11 のとおり、確定前に別セッションでの fact-check が必要
- Must 3件目(HTTPS配信手段)は「外部公開はスコープ外」という仕様の意図自体は正しいため、修正の主眼は「検証をどう成立させるかを書くこと」であり、外部公開の採否そのものはユーザー承認事項として未決に積む形を推奨する
