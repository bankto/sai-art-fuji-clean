## 総評(3行以内)

上流(D1確定・アイデア・決定メモ)との整合、スコープの In/Out、エッジケースの網羅性は全体として高い水準にある。マイルストーン粒度もM1が最小の縦切りになっており妥当。ただし「デモ位置を表示」ボタンの表示条件が仕様書内で矛盾しており、実装・検証の判定基準がぶれるため条件付き承認とする。

## 指摘事項

### Must(必ず対応)

- [x] **デモ位置ボタンの表示条件が仕様書内で矛盾している。** 根拠: 4-6の1つ目の挙動は「開発時、およびAPI非対応・権限拒否・取得失敗・施設範囲外のときに『デモ位置を表示』ボタンを表示する」とあり、条件付き表示(通常の成功状態では非表示)と読める。一方で同じ4-6の3つ目は「実GPS取得後も、検証目的でデモ位置へ切り替えられる」とあり、成功状態でもボタンが利用可能であることを前提にしている。さらに「3. 画面・ページ構成」の主な要素一覧では「デモ位置表示ボタン」を条件を付けずに常設要素として列挙しており、M1の完了条件も「デモ位置を選ぶと図上マーカーと『デモ位置』が表示される」とGPS機能実装前(M1時点)から無条件にボタンが使える前提になっている。この3箇所が一致しないため、実装者・レビュー担当のどちらがM2/M3の受け入れ基準を判定してよいか決められない。修正案: 「デモ位置を表示/切り替え」ボタンは状態によらず常時表示する(開発・検証用の恒常導線とする)、と1文に統一し、4-6の1つ目の条件列挙は「特に次の状態では案内として強調表示する」のような補助的表現に書き換える。あるいは逆に、成功状態では非表示にする方針で統一するなら、3つ目の「実GPS取得後も切り替えられる」の記述を削除する。どちらを採るかは実装の使い勝手に関わる判断のため、反映時にCodexが選択し、自律判断ログに理由を残すこと。
  - 対応: 状態によらず常時表示する方針で画面構成・4-6・M1を統一し、異常状態では案内とボタンを強調すること、および判断理由を自律判断ログへ追記した。

### Should(対応推奨)

- [x] **基準点が3点を超えた場合のアフィン変換の算出方法が未定義。** 根拠: 4-3は「基準点は最低3点の非一直線な組とし…2次元アフィン変換係数を算出または事前生成する」とあるが、3点ちょうどなら一意に解けるのに対し、4点以上(過剰決定系)の場合の解法(最小二乗など)が書かれていない。初回実装が仮データ3点のみで進むなら問題ないが、現地キャリブレーションで分散点を追加する運用（未決事項に記載あり）を見込むなら、実装前に「3点固定」か「4点以上は最小二乗で近似」かを明記した方が、後工程(基準点追加時の再実装)を防げる。
  - 対応: 基準点が3点を超える場合は、全基準点に対する最小二乗法でアフィン変換係数を近似すると4-3へ明記した。
- [x] **GitHub Pagesワークフロー(`pages.yml`)への追加点が仕様書に明記されていない。** 根拠: 現行の `.github/workflows/pages.yml` は `paths:` トリガーに `apps/camera-ar-gomi-demo/**` のみを指定しており、Node セットアップの `cache-dependency-path` も1件目アプリ専用になっている。「6. 技術構成」の「既存のGitHub Pagesワークフローへ…サブパス配置を追加する想定とする」は方向性のみで、`paths:` トリガーへの新アプリパス追加や、ビルドステップ・`public/local-map-gps-demo/` へのコピー処理の追加が必要という具体を明記していない。M3の完了条件「Pagesの想定サブパスで全アセットを解決した静的成果物を生成できる」の検証はローカルプレビューに限定されており実害はないが、実装報告時にワークフロー変更が漏れないよう一文加えることを推奨する。
  - 対応: `paths:` への新アプリ追加、build、`public/local-map-gps-demo/` への生成物コピーを6章の技術構成へ明記した。
- [x] **Node/npmバージョン固定方式が既存アプリ(camera-ar-gomi-demo)と不揃い。** 根拠: 仕様書「6. 技術構成」は `.nvmrc` + `package.json` の `packageManager` フィールドでの固定を求めているが、既存の `apps/camera-ar-gomi-demo/` にはどちらも存在せず、CIも `actions/setup-node@v6` の `node-version: 24` 固定のみで `.nvmrc` を参照していない。新方式自体は妥当だが、リポジトリ内で運用が割れる。実装時にどちらかへ揃える(新方式を1件目にも遡及するか、新方式を見送るか)方針をCodexの自律判断ログに残すよう明記すると後工程で迷わない。
  - 対応: 既存アプリと同じCIの `node-version: 24` 方式に統一し、新アプリの `.nvmrc` / `packageManager` を必須としない理由を自律判断ログへ追記した。

### Could(任意・改善提案)

- [ ] 4-1で重ね合わせレイヤーの実装手段として「HTML/SVGまたはCanvas」と両論併記になっている。テキスト表示・キーボード操作・タップ領域といった非機能要件(アクセシビリティ)を素直に満たしやすいのはDOM要素の絶対配置(HTML)なので、実装時の第一候補として明記してもよい(必須ではない)。
- [ ] 精度不足時の「設定値」(4-2)の具体的な閾値がどこにも書かれていない。実測前の暫定値(例: 30m)をコメントとして仮置きし、未決事項の実測後確定と紐づけておくと、M2のunit testが書きやすくなる。
