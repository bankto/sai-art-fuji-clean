# ワークフロー: 仕様書を元に実装する(担当: Codex)

確定済みの仕様書(指定が無ければ `docs/specs/` から確認)を元に、`apps/` 配下に実装する。
通常は Cursor(オーケストレーター)から `codex exec` でマイルストーン単位で呼び出される。

## 手順

1. 対象の仕様書と `.cursor/rules/apps.mdc`(実装ルール)を読む
2. `apps/プロダクト名/` フォルダが無ければ作成し、仕様書の技術構成どおりにセットアップする(README と `package.json` を含める)
3. 指定されたマイルストーン(指定が無ければ未完了の先頭)を実装する
4. lint・ビルドが通ることを確認する。未検証の項目は最後に「未検証」と明記して報告する
5. 実装した内容・変更ファイル・検証結果を簡潔に出力する(オーケストレーターが読む)

## レビュー指摘の修正を依頼された場合

1. 指定されたレビューレポート(`docs/reviews/`)を読む
2. Must 指摘を修正する。Should/Could は指示された場合のみ対応する
3. 対応した項目はレポートのチェックボックスを埋め、対応内容を1行添える
4. 指摘に同意できない場合は無理に修正せず、レポートの該当項目に反論理由を書く

## 注意

- 仕様に無い機能を勝手に足さない。必要だと思ったら「未決事項」として出力に含める
- 新しいライブラリの導入が必要な場合は理由を出力に明記する(承認はオーケストレーター経由でユーザーが行う)
- `docs/status.md` の更新はオーケストレーターが行う
