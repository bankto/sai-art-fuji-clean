# ワークフロー: 仕様書を元に実装する(担当: Codex)

確定済みの仕様書(指定が無ければ `docs/specs/` から確認)を元に、`apps/` 配下に実装する。
通常は Cursor(オーケストレーター)から `codex exec` でマイルストーン単位で委譲されて実行する(全フェーズ共通の委譲手順は `docs/workflows/orchestrate.md`)。単体実行でも手順は同じ。

## 手順

1. 対象の仕様書と `.cursor/rules/apps.mdc`(実装ルール)を読む
2. `apps/プロダクト名/` フォルダが無ければ作成し、仕様書の技術構成どおりにセットアップする(README と `package.json` を含める)
3. 指定されたマイルストーン(指定が無ければ未完了の先頭)を実装する
4. lint・ビルドが通ることを確認する。未検証の項目は「未検証」と明記する
5. UIや操作を変更した場合は、共通規約(a11y・キーボード操作・フォーカス制御・既存命名/構成)への対応を自己チェックする。画面操作が関係する場合は、可能な範囲でブラウザ(Playwright MCP など)で確認する
6. **`docs/reports/_template.md` に沿って `docs/reports/YYYY-MM-DD_対象名_mN.md`(作業報告)を作成する**。実装時の判断・具体的な変更内容・動作確認を書き、既存差分の列挙だけで済ませない
7. 実装した内容・変更ファイル・検証結果を簡潔に出力する(オーケストレーターが読む)

## 自律判断・停止条件

- 判断に迷ったら `docs/workflows/orchestrate.md`「自律実行ルールと STOP 条件」の優先順位に従う。低リスクな点は自律的に仮決定し、理由を作業報告の「自律判断ログ」に書く
- 質問で止まってよいのは STOP 条件に該当する場合だけ(秘密情報・認証・権限・課金・破壊的操作・仕様矛盾・完了不能・同じ問題を2回以上直しても build/lint/test が通らない・スコープ外機能の追加)

## レビュー指摘の修正を依頼された場合

1. 指定されたレビューレポート(`docs/reviews/`)を読む
2. Must 指摘を修正する。Should/Could は指示された場合のみ対応する
3. 対応した項目はレポートのチェックボックスを埋め、対応内容を1行添える
4. 指摘に同意できない場合は無理に修正せず、レポートの該当項目に反論理由を書く

## 注意

- 仕様に無い機能を勝手に足さない。必要だと思ったら「未決事項」として出力に含める
- 新しいライブラリの導入が必要な場合は理由を出力に明記する(承認はオーケストレーター経由でユーザーが行う)
- `docs/status.md` の更新はオーケストレーターが行う
