# ワークフロー: 仕様書を作成する(担当: Codex)

採用されたアイデア(指定が無ければ `docs/ideas/` のステータス「採用」から確認)を仕様書に落とし込む。
通常は Cursor(オーケストレーター)から `codex exec` で委譲されて実行する(全フェーズ共通の委譲手順は `docs/workflows/orchestrate.md`)。単体実行でも手順は同じ。

## 手順

1. 対象のアイデアシートと元リサーチを読む
2. 前提条件(スコープ・技術スタックの希望など)が指示に含まれていればそれに従う
3. `docs/specs/_template.md` に沿って `docs/specs/YYYY-MM-DD_<slug>.md` を作成する(slug は元アイデアと揃える。ASCII のみ)
4. 技術スタックは選定理由とセットで書く。複数候補で迷う場合は比較を書いた上で推奨案を選ぶ
5. 完了条件は判定可能な形で書く(「いい感じに」等の曖昧表現は禁止)。各マイルストーンには完了条件とセットで検証方法(何をどう操作・実行したら完了と言えるか)を書く
6. マイルストーンは大きく作ると失敗しやすいため極力小さく分け、各マイルストーンを「1回の委譲で完了でき、単独で動作確認できる粒度」にする。マイルストーン1は最小限動くもの(画面1枚+コア機能1つ程度の縦切り)とし、以降のマイルストーンで積み増す
7. 判断できなかった点・ユーザー確認が必要な点は「未決事項」に必ず書く(非対話実行のためユーザーには質問できない)

## 自律判断・停止条件

- 判断に迷ったら `docs/workflows/orchestrate.md`「自律実行ルールと STOP 条件」の優先順位に従う。低リスクな点は自律的に仮決定し、理由を「自律判断ログ」に書く(ユーザー確認が必要な点は「未決事項」へ)
- 質問で止まってよいのは STOP 条件に該当する場合だけ。単体実行(対話)でも、着手後は質問で止まらず最後まで完走する

## レビュー指摘の反映を依頼された場合

1. 指定されたレビューレポート(`docs/reviews/`)を読む
2. Must 指摘を仕様書に反映する。Should/Could は指示された場合のみ対応する
3. 対応した項目はレポートのチェックボックスを埋め、対応内容を1行添える
4. 指摘に同意できない場合は無理に反映せず、レポートの該当項目に反論理由を書く

## 注意

- 仕様書の「確定」への変更はユーザー承認後にオーケストレーターが行う。勝手に確定にしない
- スコープ外(やらないこと)を必ず明記する
