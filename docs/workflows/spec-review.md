# ワークフロー: 仕様書レビュー(担当: Claude Code)

作成された仕様書をレビューし、改善提案を含むレビューレポートを作成する。
通常は Cursor(オーケストレーター)から `claude -p` で委譲されて実行する(全フェーズ共通の委譲手順は `docs/workflows/orchestrate.md`)。単体実行でも手順は同じ。

## 手順

1. 対象の仕様書(`docs/specs/`)と、その元アイデア(`docs/ideas/`)・元リサーチ(`docs/research/`)を読む
2. 次の観点でレビューする
   - 上流との整合: アイデア・リサーチの意図が仕様に反映されているか
   - スコープ: やること/やらないことが明確か、暗黙のスコープ漏れがないか
   - 完了条件: 判定可能な形で書かれているか(曖昧表現がないか)。各マイルストーンに検証方法(何をどう操作・実行したら完了と言えるか)が書かれているか
   - マイルストーン粒度: 1マイルストーンが大きすぎず、1回の委譲で完了できる粒度か。マイルストーン1が最小限動くもの(縦切り)になっているか。粒度が大きすぎる場合は分割案を添えて指摘する
   - 画面・機能: 抜け漏れ、矛盾、エッジケースの考慮不足がないか
   - 技術選定: 選定理由が妥当か、過剰・過小でないか。より良い代替案があれば提案する
   - 非機能要件: 対応デバイス・アクセシビリティ・パフォーマンスの記載があるか
   - 未決事項: 放置するとまずい未決がないか
3. `docs/reviews/_template.md` に沿って `docs/reviews/YYYY-MM-DD_<slug>_spec-review_rN.md` を作成する(slug は対象ドキュメントと揃える。`N` は同じ対象のレビューラウンド。ASCII のみ)
4. 判定(承認 / 条件付き承認 / 要修正)を明記する。`## 判定` の直下は判定値1行だけにする(オーケストレーターが機械的に読むため)

## 注意

- 仕様書本体は直接修正しない。指摘と提案はレポートに書く(反映は Codex の役割)
- 指摘には根拠と具体的な修正案を添える。「もっと詳しく」のような曖昧な指摘はしない
- 改善提案(Could)も歓迎。ただし Must との区別を明確にする
- 質問で止まらない(委譲時は非対話で質問できず、単体実行でも同じ)。判断に迷った点・確認したい点はレポートの「補足」に書いて完走する
