# ワークフロー: リサーチまとめサイトを生成・更新する(主担当: Codex)

リサーチ成果を見やすくまとめたDB型の一覧サイトを `research-site/` に構築・更新する。
通常は Cursor(オーケストレーター)から `codex exec` で委譲されて実行する。単体実行でも手順は同じ。

## データの流れ(2026-07-03 ユーザー決定)

```
docs/research/(md・調査の完全記録)
  → 04_decision で採用確定
  → CSV生成(Cursor)
  → Google スプレッドシート(公開用DB・人間も編集可)← ユーザーがインポート
  → サイトはビルド時にシートから取得して静的生成
```

- スプレッドシート: https://docs.google.com/spreadsheets/d/1YJvTTgZVr9lKFffyKpsqkbbNWuNzyMDlao0iZhgF8t0/
- 列構成: 国&地域 / タイトル / URL / 概要(数行) / カテゴリ(類型) / 検証状態
- サイトのデータソースは**シートを正とする**。md から直接サイトを作らない(シートを経由することで人間の編集・追記を反映できる)

## 前提(着手条件)

1. `03_fact_check.md` が完走している(判定記入済み)
2. `04_decision.md` でユーザーが採用事例を確定している
3. CSVがシートにインポート済みである

前提が揃っていない場合は着手せず、現在のステップ(検証・判断・インポート)を案内する。

## 手順

1. シートが「ウェブに公開」(CSV形式)されているか確認する。未公開ならユーザーに公開手順を案内する
2. `research-site/` を初期構築する
   - ビルド時にシートの公開CSVを取得し、静的HTMLを生成する構成(シンプル優先。重いフレームワークは使わない)
   - トップ: 全事例の一覧。国・地域とカテゴリでの絞り込み、キーワード検索(ページ内JS)
   - 各事例: タイトル / 国&地域 / 概要 / 出典URLリンク / 検証状態
3. 検証状態の扱い: fact-check で「No判定・使用不可」となった数値・主張は表示しない。「要確認」の事例はその旨を表示する
4. レスポンシブ対応(スマホで読める)を確認する
5. 更新時(シートに行が増えた等)は再ビルドで反映する。更新手順を README に書く
6. `docs/status.md` の Phase 2 を更新する

## 注意

- サイトはあくまで閲覧用。シートに無い情報を足さない
- デザインは読みやすさ最優先。装飾よりも絞り込み・検索性を重視する
- 出典リンクはサイト上でも辿れるようにする

## オーケストレーション上の位置づけ

全体の指揮と委譲手順は `docs/workflows/orchestrate.md`(Phase 2 の節)が正本。分担は次のとおり。

- **Cursor**: 着手条件の確認、デプロイ(GitHub Actions)・公開サイトの動作確認、`docs/status.md` 更新
- **Codex**: `research-site/` の HTML/CSS/JS の作成・修正
- **Claude Code**: 要件との整合レビュー(必要時。レポートは `docs/reviews/`)
- **ユーザー**: Google スプレッドシートの編集・公開設定、Apps Script のデプロイ、GitHub Variables / Pages の設定(AI が実施できない手動作業。オーケストレーターがチェックリスト化して依頼する)
