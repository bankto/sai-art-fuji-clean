# ワークフロー: 実装レビュー(担当: Claude Code)

実装されたコードを仕様書と突き合わせてレビューし、レビューレポートを作成する。
通常は Cursor(オーケストレーター)から `claude -p` で委譲されて実行する(全フェーズ共通の委譲手順は `docs/workflows/orchestrate.md`)。単体実行でも手順は同じ。

## 手順

1. 対象の仕様書(`docs/specs/`)・作業報告(`docs/reports/`)・実装(`apps/対象/`)を読む。**作業報告の記載だけで判断せず、対象ファイルを直接確認する**
2. 次の観点でレビューする
   - 仕様との整合: 仕様書の機能・完了条件を満たしているか。仕様に無い機能が勝手に足されていないか
   - バグ: ロジックの誤り、エッジケースの考慮漏れ、エラーハンドリング不足
   - セキュリティ: シークレットのハードコード、XSS などの基本的な脆弱性
   - アクセシビリティ・共通規約: セマンティクス、キーボード操作、フォーカス制御、aria 属性、既存命名/構成との一貫性
   - 可読性・保守性: 過剰な複雑さ、重複、命名
3. 可能であれば lint・ビルドを実行して結果を確認する(コードの修正はしない)。画面操作が関係する場合は、可能な範囲でブラウザ(Playwright MCP など)で確認し、結果または未確認の理由をレポートに記録する
4. `docs/reviews/_template.md` に沿って `docs/reviews/YYYY-MM-DD_対象名_review.md` を作成する
5. 判定(承認 / 条件付き承認 / 要修正)を明記する。`## 判定` の直下は判定値1行だけにする(オーケストレーターが機械的に読むため)

## 指摘の重要度

- **ユーザー操作に関わる a11y 規約違反、または過去のフェーズで同種の指摘が出ている共通規約違反は、Should ではなく Must(必ず対応)として扱う**
- それ以外は根拠と修正案を添えて Must / Should / Could に分類する

## 注意

- コードは直接修正しない。指摘と修正案はレポートに書く(反映は Codex の役割)
- 指摘には該当ファイル・該当箇所を明記し、根拠と修正案を添える
- 動かして確認できたこと・できなかったこと(未検証)を区別して書く
