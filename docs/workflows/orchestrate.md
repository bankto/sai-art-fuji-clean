# ワークフロー: AI オーケストレーション(全フェーズ共通・担当: Cursor)

Cursor(Agents Window)は**全フェーズ**でオーケストレーターとして動く。自分で大きな作成・実装はせず、タスクを分解して Codex(作成)と Claude Code(調査・検証・レビュー)に CLI で委譲し、検証・ユーザー報告・`docs/status.md` 更新を担う。

2026-07-05 改定で Phase 4〜5 限定から全フェーズ共通に拡張(経緯: `docs/decisions/2026-07-05_全フェーズオーケストレーション改定.md`)。

## 常時の役割分担

| ツール | 常時の役割 |
|---|---|
| Cursor | オーケストレーター。タスク分解・委譲・進行管理・検証・報告・status 更新。壁打ち等のユーザー対話も担う |
| Codex | 作成担当。ドキュメント執筆、コード/サイト実装、CSV生成、レビュー指摘の反映、データ整形 |
| Claude Code | 調査・検証・レビュー担当。リサーチ収集(専門調査員)、ファクトチェック(収集とは別セッション)、仕様/実装レビュー |

## オーケストレーターの共通手順(全フェーズ)

1. **読む**: `AGENTS.md` / `docs/status.md` / 対象フェーズの `docs/workflows/*.md` を読み、現在地と前提を把握する
2. **分解**: 依頼をタスクに分解し、下の「ルーティング表」で担当を決め、タスクごとに難易度(軽量 / 標準 / 高難度)を見立てる
3. **委譲**: CLI テンプレで委譲する。委譲プロンプトには**3点セット(読むファイル・出力先・前提)を毎回必ず含める**(委譲先はこの会話の文脈を一切持っていない)。**Codex への委譲は effort を毎回明示する**(「モデル・effort の自動判断」)
4. **検証**: 成果物はファイルの実体を開いて確認する。コードは lint・ビルドも実行する。鵜呑みにしない
5. **報告**: 結果を要約してユーザーに報告する。**ユーザー承認が必要な判断ポイントでは必ず停止して聞く**。区切りの最終応答には「完了・次のタスク・セッション」を含める(「作業の区切りごとの最終応答」)
6. **記録**: `docs/status.md` を更新する(成果物1つ完成ごと。セッション末にまとめない)

## 委譲に使う CLI(PowerShell / macOS ターミナル共通)

```powershell
# Codex(作成・修正反映)。--sandbox workspace-write でファイル編集を許可(旧 --full-auto は非推奨)。effort は難易度に応じて毎回明示する
codex -c model_reasoning_effort=<medium|high|xhigh> exec --sandbox workspace-write "<指示>"

# Claude Code(調査・検証・レビュー)。acceptEdits で docs/ 配下への書き込みを許可
claude -p "<指示>" --permission-mode acceptEdits

# Claude Code(出力だけ欲しい場合: アイデア発散・相談など。ファイルを書かせない)
claude -p "<指示>"
```

- **3点セット必須**: どの委譲でも「①読むファイル(該当 workflow + 対象ファイルのパス) ②出力先 ③前提(決定済み事項・制約)」をプロンプトに含める。前提には「モデル: 既定 / effort: <指定値>」の1行も含める(委譲先が成果物の metadata に記録する。「モデル・effort の自動判断」参照)
- 委譲先は非対話で、ユーザーに質問できない。判断できない点は「未決事項として成果物に書く」よう指示に含める
- `claude -p` / `codex exec` は呼び出しごとに新しいセッション。前回の委譲内容は覚えていない。**収集と fact-check を同じ呼び出しに混ぜない**(別呼び出しに分ければ「別セッションで検証」ルールを満たす)
- 実行には数分かかることがある。完了を待って出力とファイルの実体を確認する
- 長い指示・日本語を多く含む指示は、全文を一時ファイル `_prompt.md`(`.gitignore` 済み)に書き、コマンドには短い参照だけを渡す(シェルのエスケープ事故・PowerShell の誤パースを防ぐ。`_prompt.md` には従来どおり3点セットを書く)。例: `codex -c model_reasoning_effort=<effort> exec --sandbox workspace-write "_prompt.md の指示に従って作業してください"`。使用後は削除する

### 非対話・ハング対策

委譲先が承認待ち・stdin 待ちで止まらないよう、次を守る。

- 起動はすべて非対話。人間は `codex` / `claude` にログイン済みであることを前提とする
- **Windows + Cursor Shell では stdin 待ちを避けるため `cmd /c` で包み、末尾に `< nul` を付けて stdin を閉じる**。例: `cmd /c "codex -c model_reasoning_effort=high exec --sandbox workspace-write \"_prompt.md の指示に従って作業してください\" < nul"`。長い指示・日本語入りの指示は `_prompt.md` に分離して短い参照だけを渡す(前節。日本語入りの長い引数は PowerShell に誤パースされることがある)
- **macOS では通常ラッパー不要**。stdin 待ちのハングが疑われるときだけ、コマンド末尾に `< /dev/null` を付けて再実行する(例: `codex -c model_reasoning_effort=high exec --sandbox workspace-write "<指示>" < /dev/null`)
- 承認・サンドボックス系の Codex フラグ: `-a never`(承認を求めない) / `--sandbox workspace-write`(ファイル編集を許可。**旧 `--full-auto` は非推奨**で警告が出るため使わない)。`cmd` 内で余計なエスケープや引用符を足さない
- 書き込みを許すレビュー委譲は `--permission-mode acceptEdits`。**モデルと effort は次節「モデル・effort の自動判断」で委譲時に決める**(固定しない)
- `--dangerously-bypass-approvals-and-sandbox` / `--dangerously-skip-permissions` は、隔離済み環境で明確に必要な場合以外は使わない
- **15分程度応答がなければハング扱い**。stdin 待ちが疑われる場合はプロセスを止め、stdin を閉じた形(Windows: `cmd /c "... < nul"` / macOS: 末尾に `< /dev/null`)で再実行する。同じ形式でも再度ハングするなら STOP してユーザーに報告する
- `cd` と出力リダイレクト(`>` / `>>`)を組み合わせた複合コマンドは承認で止まりやすいので避ける。ファイル作成・編集はシェルのリダイレクトでなく委譲先の編集ツールにやらせる
- **Cursor Auto-review / Run Mode の Shell 承認待ちは Cursor UI の権限確認であり、フェーズ進行可否の確認ではない**。ここまでに定義した `codex` / `claude` / `npm` / `git` コマンドなら承認して続行する。危険操作・想定外コマンドなら STOP する

## モデル・effort の自動判断(全フェーズ共通)

委譲のたびに、オーケストレーターがタスクの難易度を見て**モデルと effort をその場で決める**。固定の「フェーズ別モデル表」は作らない。原則は「**モデルは常に最善(各CLIの既定=最上位)を使い、effort を難易度で上げ下げする**」。

**前提(最初に1回だけ)**: `codex` / `claude` の既定モデルを、各CLIで使える最上位モデルに設定しておく。こうしておけば、オーケストレーターは通常モデル名を指定せず既定に任せるだけで「その時の最善のモデル」を自動的に使う。**バージョン付きのモデル名はこのドキュメントにも設定ファイルにも直書きしない**(モデル名は変わるため、CLIの既定とエイリアスに追従させる)。

難易度の見立て → effort(Codex)/ モデル:

| 難易度 | 例 | Codex の effort | モデル |
|---|---|---|---|
| 軽量 | typo・設定1行・リンク修正・単純な整形・report-only 再生成 | `medium` | 既定でよい(意図的に落とすなら `-m` で軽量モデル) |
| 標準 | リサーチノート・アイデアシート・仕様書の作成、CSV生成、一般的な実装・レビュー | `high` | 既定(=最上位) |
| 高難度 | 認証・権限・決済・複合API・データ移行・並行処理・セキュリティ、仕様全体の整合レビュー、2回目以降の再実行 | `xhigh` | 既定(=最上位)。レビューは実装より一段強くしてよい |

- **Codex 委譲**: effort は `-c model_reasoning_effort=<medium|high|xhigh>` で渡す(`exec` の前に置く)。**このフラグは省略しない**(省略すると難易度判断が反映されず CLI 既定に落ちる)。モデルは基本 `-m` を付けず既定に任せる。軽量タスクで意図的に軽いモデルへ落とすときだけ `-m` を付ける
- **`xhigh` のフォールバック**: CLI・モデルの版によっては `xhigh` を受け付けず起動時エラー(`unknown variant`)で即終了する。エラーになったら `high` に落として再実行し、CLI の更新を人間作業として報告する
- **Claude 委譲**: モデルは基本 `--model` を付けず既定に任せる。高難度レビューで明示的に上げたいときだけ `--model` をエイリアスで付ける(バージョン付きモデル名は使わない)。Claude 側に effort 相当のフラグは使わない
- **迷ったら一段上を選ぶ**(品質優先)。ただし effort を上げると実行時間は延びるので、軽量タスクにまで `xhigh` は使わない
- **使用モデル・effort の記録(作業後・毎回)**: 委譲プロンプトの前提に「モデル: 既定 / effort: <指定値>」を1行含め、委譲先は成果物冒頭の「使用AI / モデル / effort」行に転記する(前提に無ければ「不明(既定)」と書く)。オーケストレーターは成果物の検証時に記録漏れがないか確認し、`docs/status.md` の決定事項ログ(ラウンド記録)にも「誰に・何を・モデル/effort」を含めて記録する。モデルは既定に任せた場合「既定」でよい(実モデル名が分かれば併記)。高難度で意図的に上げ下げした判断の**理由**は従来どおり「自律判断ログ」に残す

高難度な実装の委譲例(effort=xhigh、モデルは既定=最上位に任せる):

```powershell
# Windows(Cursor Shell)
cmd /c "codex -c model_reasoning_effort=xhigh exec --sandbox workspace-write \"docs/workflows/implement.md を読み、docs/specs/<対象>.md のマイルストーン<N>を apps/<プロダクト名>/ に実装する\" < nul"
```

```sh
# macOS(ラッパー不要。ハング時のみ末尾に < /dev/null)
codex -c model_reasoning_effort=xhigh exec --sandbox workspace-write "docs/workflows/implement.md を読み、docs/specs/<対象>.md のマイルストーン<N>を apps/<プロダクト名>/ に実装する"
```

以降の Phase 別フローの委譲コマンド例では、読みやすさのため `cmd /c "... < nul"` ラッパー(前節「非対話・ハング対策」)のみ省略し、effort は代表的な値で記載している。実際の委譲では例の値を機械的に使い回さず、この節の判断で effort を毎回決め直し、Windows + Cursor Shell ではラッパーで包んでから実行する(macOS はラッパー不要)。

## 委譲ファースト(オーケストレーターの作業境界)

オーケストレーターの作業抱え込みは体制の破綻要因。自分で手を動かす前に、次の境界で機械的に判定する。

- **Cursor が自分で書いてよいファイル**: `docs/status.md` / 壁打ち・ヒアリングのメモ / `docs/decisions/` / 委譲用の一時プロンプトファイル(`_prompt.md`)のみ
- それ以外の成果物(リサーチノート・アイデアシート・仕様書・実装コード・作業報告・レビュー)は、**分量が少なくても自分で書かず委譲する**。「委譲するより自分でやる方が速い」は理由にしない(役割分担と検証可能性が崩れるため)
- 例外は「軽微な修正」(ルーティング表)= 1ファイル・10行以内の機械的修正まで。超える・迷ったら委譲する
- 機械的な Git・デプロイ操作(commit / push / 確立済みデプロイ / 公開確認)は逆に Cursor 自身が行い、委譲しない
- CLI 失敗時も無断で肩代わりしない(「CLI 失敗時・役割逸脱時の扱い」)

## 自律実行ルールと STOP 条件(全フェーズ共通)

質問で作業を止めすぎないための共通ルール。委譲プロンプトにも要約して含める。

- 各AI(委譲先)は、原則として作業を止めず担当範囲の完了まで進める
- 判断に迷ったら次の優先順位で決める:
  1. `docs/decisions/` の決定事項・`docs/status.md` の決定事項ログ
  2. `AGENTS.md` の運用ルール
  3. 対象フェーズの上流ドキュメント(仕様書 / アイデアシート / リサーチノート / レビュー)
  4. 既存コード・既存ドキュメントの書き方
  5. 最小変更で、後から戻しやすく、セキュリティ上安全な選択
- 低リスクで後から変更しやすい判断は自律的に仮決定し、理由を成果物の「未決事項」「自律判断ログ」に記録する
- **オーケストレーターは STOP 条件に該当しない限り、ステップ間でユーザー確認を挟まない**。レビューが承認なら次工程(反映・実装・再レビュー)へ自律的に進める
- **ユーザー承認が必要な判断ポイント(下表)と STOP 条件だけは必ず止まる**

STOP 条件(委譲先・オーケストレーター共通。該当したら進めずユーザーに報告):

- 秘密情報・認証・権限・個人情報・課金・外部サービス契約に関わる
- データ削除・破壊的操作・強制 push・履歴改変が必要
- 既存の要件・仕様・意思決定・セキュリティ方針と明確に矛盾する
- 完了条件を満たせない
- 同じ問題を2回以上修正しても build / lint / test が通らない
- 必要な成果物(作業報告など)が欠落し、再実行しても埋まらない
- 委譲コマンドが繰り返しハングする(前節「非対話・ハング対策」参照)
- スコープ外の機能追加が必要になる

## レビュー判定の読み取り(機械的に扱う)

- オーケストレーターはレビューレポートの `## 判定` 直下の1行だけを機械的に読む。品質の再判定・本文からの推測・修正方針の改変はしない
- 判定値と次アクション:
  - `承認` → そのまま次工程へ進む
  - `条件付き承認` → Must 指摘の反映を Codex に委譲 → 再確認
  - `要修正` → 具体的な修正方針を確認し、Codex に反映を委譲(該当フェーズの手順4)
- 再レビュー→反映は最大2ラウンド。収束しなければ論点を整理してユーザーに判断を仰ぐ

## Git 運用

- **commit / push はオーケストレーターだけが行う**。Codex・Claude Code は commit / push しない(レビュアーは判定と修正方針をレポートに書くだけ)
- commit するのはレビューが `承認`(または Must 反映済み)の区切り。`要修正`・対応中・WIP 状態では commit しない
- フェーズ成果物(リサーチノート・アイデアシート・仕様書など)とそのレビュー/検証が承認された区切りで commit する
- 実装はマイルストーン・作業報告・実装レビューが承認された区切りで commit する
- 運用ルール(AGENTS / README / workflows / templates / decisions)の変更は、フェーズ作業とは別コミットにする
- commit 前に `git status --short` と `git diff --stat` を確認し、対象外の変更・`.env`・秘密情報が混ざっていないか見る。混ざっていたら STOP する
- `git reset --hard`・強制 push・他者変更の revert は、ユーザーの明示依頼がない限り実行しない
- **push はオーケストレーターが自律で行う**(通常の push のみ)。commit した区切りでそのまま push し、最終応答に push 結果(リモート・ブランチ)を明記する。push を承認制にしたい案件は `docs/decisions/` にその決定を記録して運用を変える
- remote / upstream 未設定などで push できない場合は commit まで行い、push 未完了の理由と次に必要な操作を報告する

## デプロイ・公開

- **確立済みのデプロイは Cursor が自律実行する**: セットアップ済みのホスティングへの再デプロイ(push 連動の自動デプロイ、`npm run deploy` などドキュメント化済みのコマンド)は、ユーザー確認なしで実行してよい。実行後は公開 URL の表示確認までして結果を報告する
- **初回公開・デプロイ環境の新規設定はユーザー判断+人間作業**: ホスティング契約・課金プラン・ドメイン・DNS・環境変数や鍵の発行・サーバーレス等のコンソール操作は AI が代行できない(STOP 条件)。オーケストレーターは必要な作業を手順として明示し、ユーザーの完了報告を受けてからデプロイを実行する
- デプロイに失敗した場合は原因調査・修正を2回まで試み、収束しなければ STOP してエラー内容を報告する

## 作業報告(report)のチェックと report-only 再実行

- 実装(Codex)の後、オーケストレーターは `docs/reports/<日付>_<対象>_mN.md` の存在を機械チェックする
- 欠落していたら、実装コードを変更させず report だけを作らせる「report-only 再実行」を Codex に委譲する:

```powershell
codex -c model_reasoning_effort=medium exec --sandbox workspace-write "docs/reports/_template.md に従い、実装コードを変更せず、対象マイルストーンの実差分・docs/specs/<対象>.md・検証結果を確認して docs/reports/<日付>_<対象>_mN.md だけを作成する。テンプレートの各見出しを実差分・仕様・検証結果から具体的に埋め、既存差分の列挙だけで済ませないこと"
```

- 再実行しても欠落が続く場合は STOP 条件として扱う

## 作業の区切りごとの最終応答(次タスクとセッションの明示)

### 毎回の区切り(委譲1件の完了・マイルストーン完了など)

区切りごとの最終応答の末尾に、次の3項目を必ず含める(モバイル監視でもそのまま次アクションが分かるようにするため):

- **完了**: 今回終えた作業と検証結果
- **次のタスク**: 次に行う作業と担当(例: 「仕様書レビューを Claude Code に委譲」)。**人間がやるべき作業が近づいていれば AI 作業より先に明示する**: 次のフェーズ・マイルストーンまでに必要な未完了の人間作業(Google シート・Apps Script・GitHub 設定などの外部サービス作業、鍵の発行、実機確認など)を具体的な手順つきで先出しする
- **セッション**: `継続可` / `切り替え推奨(理由)`。判断基準は次節「セッションの切り替え」

### フェーズの区切り(特に実装レビュー承認)

上記に加えて、停止する前に次を明示する:

- 完了したフェーズ、レビューの最終判定、commit / push 結果
- `git status --short` の残り差分と、今回 commit しなかった理由
- 次に人間がやる具体的作業(外部サービス設定・実機確認・公開作業・不要ファイル整理など)
- 次フェーズへ入ってよいか:
  - `開始可`: 次フェーズへ進めてよい
  - `開始可(推奨作業あり)`: 進められるが、先にやると安全な人間作業がある
  - `保留`: 人間作業が終わるまで次フェーズへ入らない方がよい
- これはレビューの品質判定の再評価ではなく、外部サービス設定・未追跡差分・人間作業・STOP 条件に基づく**進行案内**として扱う

## セッションの切り替え(コンテキスト管理)

チャットが長くなるとコンテキストが肥大し、応答が遅くなり判断品質も落ちる。進行状態は常に `docs/status.md` にあるので、区切りで新しいチャットに切り替えるのが安全。次のいずれかに該当したら、最終応答の「セッション」行で `切り替え推奨` と理由を伝える。

- フェーズが完了した(成果物が「確定」した)とき
- 実装マイルストーンの commit / push が終わったとき
- 同一チャットでの委譲実行が目安10回を超えたとき
- レビュー⇄反映が2ラウンドで収束せず、論点整理してユーザー判断を仰ぐとき(新チャットで論点だけ再提示する方が判断しやすい)
- 応答が明らかに遅くなった・直前の文脈を取り違え始めたとき

切り替え手順:

1. `docs/status.md` が最新か確認する(未反映があれば先に更新する)
2. チャットにしか無い文脈(決定・数値・URL)が残っていないか確認し、あれば `docs/status.md` の決定事項ログ・該当ドキュメントに落とす
3. 最終応答で切り替えを促し、新チャットには `docs/prompts.md`「1. オーケストレーター起動」を貼るよう案内する

## Phase 1: リサーチの委譲フロー

1. **ブリーフ(Cursor)**: テーマ・モード(ライト/本格)をユーザーと壁打ちして固め、`00_brief.md` を確認する(作成の実務は Codex に委譲してもよいが、壁打ちは Cursor が担う)。**ブリーフのユーザー承認を得てから収集に進む**
2. **収集(Claude Code)**:

```powershell
claude -p "docs/workflows/research.md と docs/research/<フォルダ>/00_brief.md を読み、reference-hunter と niche-spot-scout を使って事例を収集し、docs/research/<フォルダ>/01_raw_sources.md に出典(URL+参照日)付きで追記する。前提: ブリーフは承認済み。結論は出さない。ユーザーへの質問はできないので、範囲の迷いは追記メモとして残す" --permission-mode acceptEdits
```

3. **一次情報の確認(Claude Code)**: 価格・開催日など決定に効く事実は `official-source-researcher` 指名で委譲する
4. **分析(Claude Code)**: `02_analysis.md` の作成を委譲する(収集と同じ担当でよい)
5. **検証(Claude Code・別呼び出し)**:

```powershell
claude -p "docs/workflows/research.md を読み、fact-checker として docs/research/<フォルダ>/01_raw_sources.md と 02_analysis.md を批判的に検証し、docs/research/<フォルダ>/03_fact_check.md を作成する。新しい収集はしない。前提: あなたは収集を行っていない独立の検証者である" --permission-mode acceptEdits
```

6. **判断材料の整理(Codex)**: `04_decision.md` / `05_next_actions.md` の下書きを委譲する
7. **判断(ユーザー)**: 採用/不採用/保留は**ユーザーが決める**。Cursor は論点を要約して提示し、決定を待つ
8. **整形(Codex)**: 採用確定後、CSV 生成などのデータ整形を委譲する

```powershell
codex -c model_reasoning_effort=medium exec --sandbox workspace-write "docs/research/<フォルダ>/04_decision.md を読み、採用事例を列(国&地域/タイトル/URL/概要/カテゴリ/検証状態)の CSV にして docs/research/<フォルダ>/<日付>_<slug>.csv(slug は ASCII。例: adopted-cases)に出力する。前提: 04_decision の採用マークが正"
```

## Phase 2: まとめサイトの委譲フロー

1. **前提確認(Cursor)**: `docs/workflows/research-site.md` の着手条件(fact-check 完走・04 判断済み・シート反映済み)を確認する
2. **実装・修正(Codex)**:

```powershell
codex -c model_reasoning_effort=high exec --sandbox workspace-write "docs/workflows/research-site.md と research-site/README.md を読み、<変更内容> を research-site/ に実装する。前提: データソースは Google シート経由の中継API。ビルド成果物は編集しない"
```

3. **整合レビュー(Claude Code)**: 必要に応じて(公開前・大きな変更後)委譲する

```powershell
claude -p "docs/workflows/research-site.md の要件と research-site/ の実装を突き合わせてレビューし、docs/reviews/<日付>_research-site_review.md を作成する。コードは修正しない" --permission-mode acceptEdits
```

4. **検証(Cursor)**: デプロイ(GitHub Actions)の成否、公開サイトの表示・データ取得を確認する
5. **手動作業の依頼(ユーザー)**: AI が実施できない作業はチェックリスト化してユーザーに依頼する
   - Google スプレッドシートの編集・共有/公開設定
   - Apps Script のデプロイ(Web App URL の発行)
   - GitHub Variables(`RESEARCH_SITE_DATA_API_URL` 等)の設定、Pages の有効化

## Phase 3: アイデア出しの委譲フロー

1. **壁打ち(Cursor)**: リサーチの示唆(So What)を起点に、ユーザーと対話しながら論点を整理する。対話そのものは委譲しない
2. **発散・批判(Claude Code)**: 多角的な案出しと弱点指摘を委譲し、壁打ちの材料にする

```powershell
claude -p "docs/workflows/ideate.md と docs/research/<フォルダ>/02_analysis.md の示唆を読み、アイデアを10案、各案の弱点・リスク・競合と合わせて出力する。ファイルは作成せず出力のみ。前提: <ユーザーの関心・制約>"
```

3. **整形(Codex)**: 有望案が固まったらアイデアシート化を委譲する

```powershell
codex -c model_reasoning_effort=medium exec --sandbox workspace-write "以下のアイデアを docs/ideas/_template.md に沿って docs/ideas/<日付>_<slug>.md(slug は ASCII)に整形する。上流リンクとして docs/research/<フォルダ>/ を記載する。内容: <壁打ちの結論を箇条書き>"
```

4. **採用判断(ユーザー)**: 採用・見送りはユーザーが決める。決まったら status 更新と仕様書化(Phase 4)を案内する

## Phase 4: 仕様書作成の委譲フロー

1. **事前ヒアリング(Cursor)**: 対象アイデアを読み、決めるべきこと(スコープ・画面構成・技術スタックの希望など)をユーザーに質問して固める。委譲先は非対話なのでユーザーへの質問はできない。ここで決めきれなかった点は「未決事項に書くよう」Codex に指示する
2. **作成(Codex)**:

```powershell
codex -c model_reasoning_effort=high exec --sandbox workspace-write "docs/workflows/spec.md を読み、その手順に従って docs/ideas/<対象>.md の仕様書を docs/specs/ に作成する。前提: <ヒアリング結果を箇条書き>。ユーザーへの質問はできないので、判断できない点は未決事項に書くこと"
```

3. **レビュー(Claude Code)**:

```powershell
claude -p "docs/workflows/spec-review.md を読み、その手順に従って docs/specs/<対象>.md をレビューし、docs/reviews/ にレビューレポートを作成する" --permission-mode acceptEdits
```

判定は「レビュー判定の読み取り(機械的に扱う)」に従い、`## 判定` 直下の1行だけを読む。

4. **反映(Codex)**: レポートの判定が「要修正」または「条件付き承認」なら、Must 指摘の反映を Codex に委譲する

判定は「レビュー判定の読み取り(機械的に扱う)」に従い、本文から推測せず機械的に読む。

```powershell
codex -c model_reasoning_effort=high exec --sandbox workspace-write "docs/reviews/<レポート>.md の Must 指摘を docs/specs/<対象>.md に反映する。Should/Could は対応する場合のみ。対応内容をレポートのチェックボックスに反映すること"
```

5. **収束判定(Cursor)**: 再レビュー→反映は最大2ラウンド。収束しない場合は論点を整理してユーザーに判断を仰ぐ
6. **ユーザー承認**: 仕様書・レビュー・未決事項を要約して提示し、ユーザーの承認を得てから仕様書を「確定」にする。`docs/status.md` を更新する

## Phase 5: 実装の委譲フロー

1. **事前確認(Cursor)**: 仕様書が「確定」であることを確認。マイルストーンを把握する
2. **実装(Codex)**: マイルストーン単位で委譲する

```powershell
codex -c model_reasoning_effort=high exec --sandbox workspace-write "docs/workflows/implement.md を読み、docs/specs/<対象>.md のマイルストーン<N>を apps/<プロダクト名>/ に実装する"
```

3. **作業報告の存在確認(Cursor)**: `docs/reports/<日付>_<対象>_mN.md` が作成されているか確認する。欠落していれば「作業報告(report)のチェックと report-only 再実行」に従い、実装コードを変更させず report-only 再実行を委譲する
4. **レビュー(Claude Code)**:

```powershell
claude -p "docs/workflows/implement-review.md を読み、docs/specs/<対象>.md・docs/reports/<日付>_<対象>_mN.md と突き合わせて apps/<プロダクト名>/ をレビューし、docs/reviews/ にレビューレポートを作成する" --permission-mode acceptEdits
```

判定は「レビュー判定の読み取り(機械的に扱う)」に従い、`## 判定` 直下の1行だけを読む。

5. **修正(Codex)**: Must 指摘があれば Codex に修正を委譲する(Phase 4 の手順4と同様)

判定は「レビュー判定の読み取り(機械的に扱う)」に従い、本文から推測せず機械的に読む。

6. **検証(Cursor)**: lint・ビルドが通ることを自分でも確認する。ブラウザでの動作確認が可能ならスクリーンショットを添えて報告する
7. **報告**: 実装内容・作業報告・レビュー結果・未検証項目を要約してユーザーに報告し、`docs/status.md` を更新する

## ユーザーが必ず判断するポイント(AI は勝手に進めない)

| フェーズ | 意思決定ポイント |
|---|---|
| 1. リサーチ | テーマとブリーフ(目的・範囲・判断基準)の承認 / 本格リサーチの採用・不採用・保留の最終判断(`04_decision.md`) |
| 2. まとめサイト | 初回公開の可否 / シート・Apps Script・GitHub 設定などの手動作業(AI 実施不可) |
| 3. アイデア出し | アイデアの採用・見送り |
| 4. 仕様書 | スコープと技術スタックの承認 / 仕様書の「確定」 |
| 5. 実装 | マイルストーン完了の確認 / レビューが収束しないときの論点判断 / 初回公開・リリース(確立済みデプロイの再実行と push は自律。「デプロイ・公開」参照) |
| 全フェーズ共通 | フェーズを飛ばす判断 / 新しいライブラリ・外部サービスの導入 / 費用が発生する操作 |

## 実行環境の運用メモ(Agents Window / iOS)

- 拠点は Cursor の **Agents Window**。ユーザーは iOS アプリからも指示・監視する
- **Remote Control(デスクトップのエージェントを遠隔操作)**: デスクトップ上で動くため `codex` / `claude` CLI がそのまま使える。**委譲を含む全機能が動く推奨形態**。Cursor は `/remote-control` 等でモバイルへの引き渡しが可能
- **Cloud Agent(クラウド実行)**: 実行環境に `codex` / `claude` CLI が無い・未認証のことがある。委譲コマンドが失敗したら、無理に自分で作成作業を肩代わりせず「この作業はデスクトップ(Remote Control)での再開が必要」とユーザーに報告して止まる。軽い確認・調査・status 更新など Cursor 単体で完結する作業のみ進めてよい
- モバイルからの監視を想定し、**進捗は `docs/status.md` に常に反映**する(成果物1つごと)。報告は短く、承認を求めるときは選択肢を簡潔に示す
- 長時間タスクは分割して委譲する(CLI のタイムアウトと、途中中断からの再開性のため)

## CLI 失敗時・役割逸脱時の扱い

- **CLI 失敗(未認証・タイムアウト・コマンド不在)**: エラー内容をそのままユーザーに報告し、選択肢を提示する(①認証・環境を直して再実行 ②Cursor 自身が代行 ③後回し)。**ユーザー承認なしに役割を無視して代行しない**
- **役割逸脱の依頼**: 自分の役割外の作業を求められたら、役割分担を伝えたうえでユーザーの指示に従う(`AGENTS.md` のルール)
- **委譲先の暴走**: 成果物が指示範囲を超えていたら(頼んでいないファイルの変更など)、差分を確認してユーザーに報告する。無断で握りつぶさない
- **AI 間のラウンド記録**: 誰が・何を・何ラウンド行ったか(使用モデル・effort を含む)を `docs/status.md` の決定事項ログに書く

## オーケストレーターの心得

- 委譲した作業の成果物は鵜呑みにせず、ファイルの実体・lint・ビルドで検証する
- 会話の文脈はファイルに残す(委譲先・次セッション・モバイルからの再開のため)
- 自分で手を動かすのは「軽い確認・検証・報告・status 更新・commit / push / 確立済みデプロイ」まで。作成に踏み込みそうになったら委譲に切り替える
