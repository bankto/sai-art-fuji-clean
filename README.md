# sai-art-fuji

リサーチ → まとめサイト → アイデア出し → 仕様書 → 実装 を AI と進めるプロジェクト。
あなた(人間)がやるのは**依頼・承認・手動作業・最終判断**だけ。実作業は3つのAIが分担します。

| 役割 | 担当 |
|---|---|
| 指揮(タスク分解・委譲・検証・報告) | Cursor(Agents Window) |
| 作成(文書・コード・CSV・データ整形) | Codex(Cursor から自動委譲) |
| 調査・検証・レビュー | Claude Code(Cursor から自動委譲) |
| **依頼・承認・手動作業・最終判断** | **あなた** |

---

## クイックスタート

Cursor の Agents Window を開いて、これを貼るだけ:

```
AGENTS.md と docs/status.md と docs/workflows/orchestrate.md を読んで、オーケストレーターとして動いて。作成は Codex、調査・検証・レビューは Claude Code に CLI 委譲。軽い確認と status 更新は Cursor 自身。ユーザー承認が必要な判断は止まって聞いて。
```

続けて、やりたいことを1行添える(例: 「◯◯のリサーチを始めたい」)。

**2回目以降・続きから再開するとき:**

```
docs/status.md を読んで現状を把握して、オーケストレーターとして続きを進めて。委譲ルールは docs/workflows/orchestrate.md に従って。
```

**今どこまで進んでいるかは、常に [docs/status.md](docs/status.md) を見れば分かります**(全AIが成果物1つ完成ごとに更新するルール)。

## あなた(人間)がやること

AIは以下を勝手に進めません。**この節だけ把握しておけば運用が回ります。**

### 全フェーズ共通

- やりたいことを依頼する(下の「プロンプト集」参照)
- AIが「承認してください」と止まったら判断を返す
- AIが実施できない手動作業(外部サービスのアカウント操作など)を行う
- 次の操作も承認対象: フェーズを飛ばす / 新しいライブラリ・外部サービスの導入 / 費用が発生する操作

### フェーズごとの人間タスクと注意点

| フェーズ | あなたがやること | 注意点 |
|---|---|---|
| 1. リサーチ | ① テーマを伝える ② ブリーフ(調査の設計図)を承認する ③ `04_decision.md` で採用/不採用/保留を最終判断する | ブリーフ承認前に収集は始まらない。③を放置するとフェーズが先に進まない |
| 2. まとめサイト | ① Google シートの編集・「ウェブに公開」設定 ② Apps Script のデプロイ(Web App URL 発行) ③ GitHub Variables / Pages の設定 ④ 公開OKの最終確認 | ①〜③はアカウント操作のためAIが代行できない。AIがチェックリストを出すのでそれに沿って作業する |
| 3. アイデア出し | ① 壁打ちに付き合う(相手は Cursor) ② アイデアの採用・見送りを決める | 「どれも良いね」で終わらせない。採用を明言しないと Phase 4 に進めない |
| 4. 仕様書 | ① 事前ヒアリングに答える(スコープ・画面・技術の希望) ② 仕様書を読んで承認し「確定」を指示する | 「未決事項」の欄は必ず読む。確定後の大幅変更は手戻りが大きい |
| 5. 実装 | ① マイルストーン完了を確認する ② レビューが収束しないときの論点を判断する ③ 公開・リリースを決断する | 公開前に自分の目でも動作確認する。AIの「動きます」を鵜呑みにしない |

### 判断を求められたときの返し方(例)

- 承認: 「OK、進めて」「ブリーフ承認。収集を開始して」
- 条件付き: 「◯◯だけ直してから進めて」
- 保留: 「保留。status.md に保留理由を記録しておいて」

## プロンプト集(コピペ用)

### 起動・再開

クイックスタートの2つ(基本起動/再開)を使う。スラッシュコマンド派は `/orchestrate` でも同じ。

### フェーズ別の依頼例

```
/research 廃棄物アップサイクルの国内市場          ← Phase 1 開始
/research-site カード一覧の配色を◯◯に変えたい      ← Phase 2 の修正依頼
/ideate                                        ← Phase 3 壁打ち開始
/spec 2026-07-10_アイデア名                      ← Phase 4(採用アイデアの仕様化)
/implement 2026-07-12_プロダクト名               ← Phase 5(確定仕様書の実装)
```

### 良い指示のコツ

- 「**対象(どのファイル・どの画面)+ やりたいこと + 決まっている前提**」をセットで伝えると一発で通る(AI同士の委譲でも同じ3点セットルールを使っている)
- 自分で決めたいことは先に宣言する(例: 「技術選定は私が決めるので候補比較まで」)
- 完成形が曖昧なときは「まず案を3つ、比較付きで」と発散から頼む

## Cursor iOS アプリでの運用

### 形態は2つ。Remote Control を推奨

| 形態 | 仕組み | このプロジェクトでの可否 |
|---|---|---|
| **Remote Control(推奨)** | デスクトップPCで動いているエージェントを iOS から指示・監視 | ◎ 全機能が動く(委譲CLIがPC上で実行されるため) |
| Cloud Agent | クラウド環境でエージェントが動く | △ 軽い調査・確認・相談のみ。環境に `codex`/`claude` CLI が無い場合、委譲が必要な作業は止まって報告してくる(ルール化済み) |

### 外出前のチェックリスト(デスクトップ側)

1. `codex --version` / `claude --version` が通り、両CLIがログイン済みであること
2. Cursor の Agents Window でエージェントを開始しておく(継続中セッションがあればそのまま)
3. **PCをスリープさせない**(電源設定を確認。スリープすると遠隔操作も委譲も止まる)
4. iOS アプリで同じアカウントにログインし、**通知をオンにする**(承認待ちで止まっているのに気づかないのが一番のロス)

### iOS からの操作

- 送るプロンプトはデスクトップと同じ(クイックスタート参照)
- 進捗確認: 「docs/status.md の現在地を3行で要約して」
- 承認待ちが来たら判断を返す。急がないものは「保留」でよい(帰宅後にデスクトップで差分を見てから承認する運用が安全)

### 注意点

- 長時間タスクは「マイルストーンごとに status.md を更新して報告して」と添えると、途中で切れても再開しやすい
- 大きな決断(仕様の確定・公開・費用が発生する操作)は外出先で即決しない
- 委譲失敗の報告(「CLIが使えない」等)が来たら: Cloud Agent で動いている可能性が高い。その作業は帰宅後に Remote Control / デスクトップで再開すればよい

## AI利用の Tips

### 新しいセッション・別のPCで始めるとき

文脈はすべてファイルに残る設計(会話ログに依存しない)なので、**再開プロンプト1行で続きから動きます**。

別のPCの場合は追加で:

1. このプロジェクトフォルダ一式を同期する
2. `codex` / `claude` CLI をインストールし、ログインする(確認: `codex --version` / `claude --version`)
3. あとは同じ再開プロンプトでOK

### コンテキスト(会話)がいっぱいになった・応答が鈍くなったとき

1. 今のセッションに一言: 「**ここまでの決定事項と途中状態を docs/status.md に反映して**」
2. 新しいセッションを開いて、再開プロンプトを送る

### セッションの切り替えどき(「1タスク=1セッション」の意味)

毎回の指示ごとに新セッションにするという意味ではなく、「1つの長いセッションに何タスクも詰め込まない」という意味です。

- セッションを**続ける**: 壁打ちの途中、ヒアリングへの回答中、修正の往復中など、対話が続いているとき
- セッションを**切り替える**: タスクが一区切りした / 翌日・別PCで再開する / 応答が鈍い・話が噛み合わなくなってきたとき(切替前に上記の status.md 反映の一言)

**新セッションでも文脈が消えない理由**: このプロジェクトは決定事項・進捗・成果物をすべてファイル(status.md など)に残す設計で、新しいセッションはそれを読んで文脈を再構築します。委譲される Codex / Claude Code はもともと毎回文脈ゼロで動いており、セッション切り替えも同じ仕組みに乗るだけです。
ただし**ファイルに書かれていない「会話だけのニュアンス」はセッションと一緒に消えます**。壁打ちで出た結論や好みは「ファイルに残しておいて」と言っておくと安全です。長いセッションを粘るほうがむしろ危険で、コンテキストが溢れると古い会話から静かに忘れられていきます(ファイル化された文脈は劣化しません)。

### その他のコツ

- **複数AIに同じ問いを投げる**: 意見が割れた箇所こそ論点。Phase 1 では Claude Code と Codex の並走で相補的な結果が得られた実績あり(status.md 決定事項ログ 2026-07-03)
- **承認前は自分の目でも見る**: AIの要約ではなく、ファイルの実体・実際のサイトを開いて確認する
- **AIが役割から外れた動きをしたら**: 「AGENTS.md を読み直して」の一言で戻る
- **迷子になったら**: 「今の状態と、次に取れる選択肢を整理して」と聞く
- **大きな変更の前はスナップショット**: git でコミットしてから始めると巻き戻せる(※現在このフォルダは git 未初期化。最初に `git init` と初回コミットを済ませておくことを強く推奨)

### 特定のAIに直接頼みたいとき(単体実行)

オーケストレーター経由が基本ですが、直接頼んでも成果物の置き場所・ルールは同じです。

| 頼みたいこと | 開くAI | 入口 |
|---|---|---|
| リサーチの収集・分析・検証 | Claude Code | `/research テーマ` |
| 仕様書・実装のレビュー | Claude Code | `/spec-review 対象` / `/implement-review 対象` |
| サイト実装・仕様書作成・実装 | Codex | research-site / spec / implement スキル(自然文で依頼) |
| アイデアの発散・批判 | Claude Code | `/ideate` |

```
sai-art-fuji-clean/

├── CLAUDE.md            # Claude Code 用ブリッジ(@AGENTS.md + 調査・検証・レビュー役の指定)
├── .cursor/
│   ├── rules/           # フェーズ別詳細ルール + orchestrator.mdc(常時適用の役割定義)
│   └── commands/        # Cursor 用スラッシュコマンド(/orchestrate ほか)
├── .claude/
│   ├── commands/        # Claude Code 用(調査・レビュー系)
│   ├── agents/          # リサーチ専門調査員(収集・ニッチ探索・一次情報・検証)

├── .agents/
│   └── skills/          # Codex 用(作成系: research-site / spec / implement など)
├── docs/
│   ├── status.md        # 進行状況ダッシュボード(現在地はここを見る)
│   ├── decisions/       # 運用に関する決定記録
│   ├── workflows/       # フェーズ別手順書(orchestrate.md が委譲の正本)
│   ├── research/        # Phase 1: リサーチノート
│   ├── ideas/           # Phase 3: アイデアシート
│   ├── specs/           # Phase 4: 仕様書
│   └── reviews/         # Claude Code のレビューレポート
├── research-site/       # Phase 2: リサーチまとめサイト
└── apps/                # Phase 5: 実装するWebサイト/Webアプリ
```

## ルールの仕組み(どこを直せば全体に効くか)

手順・ルールの本体は1か所に集約し、各ツールの設定はそれを参照する薄い入口です。**ルールを変えたいときは本体側だけを編集**してください。

| 層 | 本体 | 入口 |
|---|---|---|
| 全体ルール・役割分担 | `AGENTS.md` | Cursor / Codex は直接読む。Claude Code は `CLAUDE.md` 経由 |
| 委譲手順・振り分け | `docs/workflows/orchestrate.md` | `/orchestrate` ほか各コマンドが参照 |
| フェーズ手順 | `docs/workflows/*.md` | 各ツールのコマンド / スキルが参照 |
| フェーズ別詳細ルール | `.cursor/rules/*.mdc` | Cursor は自動適用。他は AGENTS.md の指示で読む |

## 各ドキュメントの雛形

- リサーチ(ライト): `docs/research/_templates/light.md`
- リサーチ(本格): `docs/research/_templates/full/00〜05`
- アイデア: `docs/ideas/_template.md`
- 仕様書: `docs/specs/_template.md`
- レビュー: `docs/reviews/_template.md`
